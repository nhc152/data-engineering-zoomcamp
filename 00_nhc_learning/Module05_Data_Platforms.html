Data Platforms giải quyết gì?

Pipeline SQL rải rác không ai biết chạy theo thứ tự nào, không có lineage, không có test, deploy bằng tay — khi có incident không biết bắt đầu debug từ đâu.

Cảnh báo: Khi pipeline chỉ là một đống script, vấn đề không nằm ở SQL khó hay Python khó. Vấn đề là không có contract vận hành: ai sở hữu output, input nào làm nó đổi, và chạy lại có an toàn không.

Bruin giải quyết gì?

Orchestration ad-hoc, assets không có dependency graph rõ ràng, không reproducible, không version-controlled, không có built-in quality checks.

Lưu ý: Bruin gom ingestion, transformation, orchestration, data quality, metadata và lineage vào cùng một project git-native.

Trước Bruin / data platform tools người ta làm gì?

  • Shell scripts chạy tuần tự: dễ hiểu lúc nhỏ, nhưng khó retry từng bước và khó biết bước nào sinh ra bảng nào.
  • Cron jobs: chạy được theo lịch, nhưng dependency graph thường nằm trong đầu người viết.
  • Notebook pipelines: phù hợp exploration, nhưng dễ mất reproducibility nếu state trong notebook không sạch.
  • Custom orchestrators: linh hoạt, nhưng team phải tự maintain scheduler, metadata, retry, lineage và quality checks.

1 Data Platforms & Orchestration ▼

Beginner: Data Platform là gì?

Data Platform là lớp công cụ giúp quản lý vòng đời dữ liệu từ ingest, transform, orchestration, quality checks, metadata đến lineage. Nó khác Data Warehouse vì warehouse là nơi lưu/truy vấn dữ liệu; khác Data Pipeline vì pipeline là một luồng xử lý cụ thể. Platform là hệ thống quản trị nhiều pipeline và nhiều assets.

Lưu ý: BigQuery/Snowflake là warehouse. Một pipeline NYC Taxi là flow xử lý. Bruin là platform/tooling giúp định nghĩa, chạy, kiểm tra và quan sát flow đó.

Beginner: Asset-based thinking vs task-based thinking

Task-based hỏi “bước nào chạy trước?”. Asset-based hỏi “bảng/view/file nào là output cần tin cậy, nó đến từ đâu?”. Asset là unit tốt hơn task vì người dùng data quan tâm output và contract của output, không chỉ quan tâm script đã chạy.

Tip: Với production, hãy review theo asset: owner, upstream, freshness, schema, quality checks, rerun behavior.

Beginner: Pipeline, Asset, Connection

  • Pipeline: nhóm assets theo schedule và config, ví dụ nyc_taxi chạy daily.
  • Asset: một file SQL/Python/YAML tạo hoặc cập nhật một bảng/view, ví dụ staging.trips.
  • Connection: cấu hình kết nối tới DuckDB, BigQuery, Snowflake, Postgres..., thường nằm trong .bruin.yml local hoặc Bruin Cloud.
Nguy hiểm: .bruin.yml có thể chứa credentials. Repo notes nhấn mạnh file này phải local và không push lên git.

Intermediate: Dependency graph (DAG)

DAG mô tả upstream/downstream của assets. Trong NYC Taxi, staging.trips phụ thuộc ingestion.trips và ingestion.payment_lookup; reports.trips_report phụ thuộc staging.trips. DAG quan trọng vì nó quyết định execution order, rerun scope và debug path khi incident xảy ra.

Intermediate: Materialization trong Bruin

Bruin patternÝ nghĩaSo với dbt
tableTạo bảng vật lý, phù hợp output cần query nhanh.Gần materialized='table'.
viewKhông lưu data, query lại khi đọc.Gần materialized='view'.
incremental / time_intervalChỉ xử lý một vùng dữ liệu, ví dụ theo pickup_datetime.Gần dbt incremental, nhưng Bruin gắn trực tiếp với run interval và orchestration.
Cảnh báo: Raw asset trong repo dùng append; nếu rerun cùng interval có thể duplicate raw data. Staging có dedup, nhưng raw vẫn phình ra nếu không kiểm soát.

Senior: Bruin trong modern data stack

Bruin nằm ở lớp platform/orchestration quanh pipeline: ingest từ source, transform bằng SQL/Python, chạy quality checks, tạo metadata/lineage, rồi serve output cho reporting hoặc downstream applications. Nó không thay thế warehouse, nhưng làm pipeline reproducible và observable hơn.

Mental model: Bruin = git-native data platform: define assets, declare dependencies, run anywhere.

2 Bruin Core Concepts ▼

Beginner: Projects, Pipelines, Assets

Project là root directory đã init bằng bruin init, chứa config và pipelines. Pipeline gom assets theo schedule/config, có pipeline.yml. Asset là một file làm việc cụ thể: ingest, transform, seed, report.

● ● ●project-shape.txt
project/
  .bruin.yml
  pipeline/
    pipeline.yml
    assets/
      ingestion/trips.py
      staging/trips.sql
      reports/trips_report.sql

Beginner: Asset types

Asset typeDùng khi nàoVí dụ repo
SQL assetTransform, aggregate, build staging/report tables.staging/trips.sql, reports/trips_report.sql
Python assetIngestion/API/file processing cần Python libraries.ingestion/trips.py
Ingestr/YAML/Seed assetLoad source hoặc static lookup theo config.payment_lookup.asset.yml

Intermediate: Variables trong Bruin

Built-in vars như start_date, end_date được tạo theo run interval. SQL dùng Jinja như {{ start_date }}; Python nhận qua environment variables như BRUIN_START_DATE, BRUIN_END_DATE hoặc custom vars qua JSON.

Tip: Variables giúp cùng một asset chạy cho nhiều interval hoặc nhiều tenant mà không sửa code.

Intermediate: Connections

Connections định nghĩa cách Bruin nói chuyện với destination/source: BigQuery, DuckDB, Snowflake, Postgres, MotherDuck... Local config thường nằm trong .bruin.yml; Bruin Cloud lưu connection trong UI/cloud secrets nhưng tên connection phải khớp với pipeline.

Senior: Bruin MCP integration

Bruin MCP cho phép AI agent trong Cursor/VS Code/Claude giao tiếp với Bruin để scaffold assets, validate pipeline, query data và giải thích pipeline logic. Workflow tốt là dùng AI để draft asset/config, rồi engineer review metadata, dependency, materialization, quality checks và chạy validate.

Nguy hiểm: AI-generated pipeline vẫn cần review như production code. Đặc biệt kiểm tra logic delete/insert, date boundaries, secrets và dependency graph.
Mental model: Bruin asset = 1 file (SQL hoặc Python) + metadata YAML block; Bruin lo dependency và execution.

Schedule syntax

Built-in schedules như daily, weekly, monthly đủ cho tutorial. Với production phức tạp, cron format kiểu 0 3 * * * giúp mô tả “3 giờ sáng mỗi ngày”. Dù dùng format nào, điều quan trọng là assets trong cùng pipeline nên có cùng cadence.

Điểm yếu: Reference config chưa có owner, alert policy, retry policy, SLA/freshness expectation. Đây là gap thường gặp khi chuyển từ tutorial sang production.
I Phần 4 — Ingestr / Seed Assets ▼

Trong notes, ví dụ “YAML ingestor asset” được mô tả ở level concept. NYC Taxi reference dùng seed asset để load lookup CSV, không có file ingestr source/destination đầy đủ trong workspace.

● ● ●payment_lookup.asset.yml
name: ingestion.payment_lookup
type: duckdb.seed
parameters:
  path: payment_lookup.csv
columns:
  - name: payment_type_id
    type: integer
    description: "Numeric code for payment type"
    primary_key: true
    checks:
      - name: not_null
      - name: unique
  - name: payment_type_name
    type: string
    description: "Human-readable payment type"
    checks:
      - name: not_null
  • type: duckdb.seed: asset load file tĩnh vào DuckDB, không phải custom Python.
  • parameters.path: source file local payment_lookup.csv.
  • columns: metadata + data quality checks chạy sau asset.
  • Ingestr asset concept: nếu dùng ingestr thật, parameters thường thể hiện source connection, source table/stream và destination table; lợi ích là không phải tự viết extraction script cho connector phổ biến.
Khi dùng ingestr thay custom script: source là connector phổ biến, mapping đơn giản, không cần business logic phức tạp. Khi cần API pagination riêng, retry tùy biến hoặc enrich dữ liệu trong code, Python asset hợp lý hơn.
Điểm yếu: Seed lookup nhỏ thì ổn, nhưng với ingestion lớn cần incremental cursor, error handling, retry, schema drift policy. Notes không có ingestr asset đầy đủ để đánh giá các config này.

/* @bruin
name: staging.trips
type: duckdb.sql

depends:
  - ingestion.trips
  - ingestion.payment_lookup

materialization:
  type: table
  strategy: time_interval
  incremental_key: pickup_datetime
  time_granularity: timestamp

columns:
  - name: pickup_datetime
    type: timestamp
    primary_key: true
    checks:
      - name: not_null

custom_checks:
  - name: row_count_greater_than_zero
    query: |
      SELECT CASE WHEN COUNT(*) > 0 THEN 1 ELSE 0 END
      FROM staging.trips
    value: 1
@bruin */

SELECT
    t.pickup_datetime,
    t.dropoff_datetime,
    t.pickup_location_id,
    t.dropoff_location_id,
    t.fare_amount,
    t.taxi_type,
    p.payment_type_name
FROM ingestion.trips t
LEFT JOIN ingestion.payment_lookup p
    ON t.payment_type = p.payment_type_id
WHERE t.pickup_datetime >= '{{ start_datetime }}'
  AND t.pickup_datetime < '{{ end_datetime }}'
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY t.pickup_datetime, t.dropoff_datetime,
                 t.pickup_location_id, t.dropoff_location_id, t.fare_amount
    ORDER BY t.pickup_datetime
) = 1

Roadmap yêu cầu không qua module mới nếu chưa tự debug ít nhất vài lỗi. Với Bruin, hãy phá ở metadata, connection, variables và concurrency vì đó là nơi incident production thường bắt đầu.

Kịch bảnTriệu chứng / lỗi dự kiếnDebug flowCách fix
1. Khai báo depends sai tên asset bruin validate báo broken dependency / asset not found trước khi run. Đọc asset name trong YAML header, chạy bruin lineage, so sánh tên thật với depends. Sửa depends thành đúng name, không dùng file path thay asset name.
2. Connection name không khớp project config Validate/run báo missing connection hoặc cannot resolve connection duckdb-default. Mở .bruin.yml và pipeline.yml, kiểm tra default_connections trỏ tới name nào. Đổi tên connection trong project config hoặc pipeline để khớp 1-1.
3. Run khi upstream asset chưa tồn tại Asset downstream fail vì table/view upstream không tồn tại, hoặc validation báo missing upstream asset. Chạy bruin lineage --asset <asset>, rồi chạy lại với --upstream. Tạo upstream asset, sửa dependency, hoặc run --asset staging.trips --upstream.
4. Variables {{ start_date }} không resolve bruin render vẫn còn literal Jinja hoặc SQL run lỗi vì date string không hợp lệ. F 1. FAILURE: Pipeline chạy 3 tiếng, fail ở transform asset lúc 2h sáng ▼

Log xem ở đâu? Nếu chạy local, xem terminal output của bruin run, rerun với scope hẹp hơn, và dùng bruin lineage để xác định upstream/downstream của asset fail. Nếu deploy Bruin Cloud, xem run detail trong UI: asset status, execution logs, quality check results và dependency graph.

Asset đang ở trạng thái gì sau khi fail? Asset fail không nên được xem là output sạch. Với table append hoặc incremental không atomic, có thể đã có partial write. Với transform table/time_interval, state có thể là dirty state nếu bước delete/insert/merge chạy một phần trước khi lỗi xảy ra.

bruin run --asset retry hoạt động thế nào? Retry bằng cách chạy lại đúng asset fail. Cách này đủ khi upstream đã sạch, lỗi là transient như network, timeout, warehouse slot tạm thiếu, hoặc bug SQL đã fix và asset có materialization idempotent. Cách này không đủ khi upstream cũng bị sai, raw table đã duplicate, schema nguồn đổi, hoặc asset fail đã để lại partial write cần cleanup/full refresh.

Fix thực tế: bắt đầu từ failed asset nếu upstream đã pass và data contract không đổi. Rerun toàn pipeline khi lỗi có thể lan từ ingestion, biến date range sai, connection sai environment, hoặc nhiều downstream phụ thuộc vào một output đã bẩn.

Khi nào dùng --full-refresh? Dùng cho incremental asset khi logic tính toán thay đổi, unique key sai trước đó, historical rows đã bị duplicate, hoặc cần rebuild lại partition/table từ source of truth. Tác động downstream là tất cả asset đọc từ table đó nên được rerun, thường dùng thêm --downstream.

● ● ●failure-recovery.sh
# 1. Inspect graph around failed asset
bruin lineage ./pipeline/pipeline.yml --asset staging.trips

# 2. Retry only failed transform when upstream is trusted
bruin run ./pipeline/pipeline.yml --asset staging.trips \
  --start-date 2022-01-01 --end-date 2022-02-01

# 3. If staging was partially written, rebuild it and its consumers
bruin run ./pipeline/pipeline.yml --asset staging.trips \
  --full-refresh --downstream \
  --start-date 2022-01-01 --end-date 2022-02-01
S 2. SCALE: Từ demo data vài MB lên TB/ngày ▼

Ingestr/YAML ingestion asset hiện tại có đủ không? Notes module có nhắc YAML ingestor ở level concept, còn NYC Taxi reference dùng Python asset ingestion.trips và seed asset ingestion.payment_lookup. Với production, chỉ dùng ingestr khi connector/source được Bruin docs hỗ trợ rõ và đã verify field syntax theo version CLI đang chạy.

Incremental ingestion: production không nên full reload hằng ngày nếu source có timestamp, cursor, primary key hoặc partition. Asset cần chỉ pull khoảng start_date/end_date, ghi vào partition/table đích bằng strategy có khả năng overwrite/merge đúng interval.

Date range variables là chìa khóa scale: cùng một asset có thể chạy daily, hourly, backfill một tháng, hoặc rerun một ngày lỗi mà không đổi code. Không có date boundary thì mọi retry dễ thành full scan/full reload.

Khi nào chuyển từ DuckDB sang BigQuery/Snowflake? Khi data vượt RAM/local disk, nhiều người cần query đồng thời, cần IAM/audit, cần partition/cluster/compute scale, hoặc pipeline trở thành SLA production. DuckDB rất tốt cho local development và dữ liệu nhỏ-trung bình, không phải production warehouse cho TB/ngày và nhiều consumer.

I 3. IDEMPOTENCY: Chạy pipeline 2 lần cùng date range có duplicate không? ▼

Ingestr asset idempotent mặc định không? Không nên giả định. Nếu asset append thẳng vào raw table, rerun cùng interval có thể duplicate. Idempotency phụ thuộc vào connector, destination strategy, unique key và cách xử lý interval.

SQL transform: view vs table. View thường idempotent vì chỉ lưu query definition. Table rebuild cũng idempotent nếu mỗi run replace toàn bộ output hoặc overwrite đúng interval. Table append không idempotent nếu không có delete/merge trước khi insert.

Incremental asset với unique key: idempotent khi merge/upsert dựa trên key ổn định, hoặc time_interval delete-and-insert đúng cùng date window. Key phải đại diện cho business identity, không phải timestamp dễ thay đổi.

Truly idempotent Bruin pipeline: chạy lại cùng code, cùng input, cùng date range sẽ tạo cùng output; không tăng row count bất thường, không đổi aggregate ngoài kỳ vọng, quality checks vẫn pass, downstream không bị nhân đôi.

● ● ●idempotency-check.sh
# Count before rerun

Q 4. DATA QUALITY: Làm sao biết data đúng sau khi pipeline chạy? ▼

Built-in checks: notes module dùng các checks như not_null, unique, non_negative trong block columns. Các check này chạy sau khi asset materialize xong và giúp bắt lỗi null key, duplicate primary key, metric âm.

Custom assertions: dùng custom_checks khi rule là business logic, ví dụ row count phải lớn hơn 0, tổng doanh thu không âm, hoặc một phân vùng phải có đủ ngày.

Source freshness: với production, thêm check so sánh MAX(updated_at) hoặc MAX(event_time) với run date. Pipeline thành công nhưng source stale vẫn là incident dữ liệu.

Row count anomaly: row count giảm 90% thường không làm SQL fail, nên cần check tương đối so với historical baseline hoặc ít nhất threshold tối thiểu theo ngày.

/* @bruin

C 5. COST & PERFORMANCE: Tránh reprocess không cần thiết ▼

Full reload vs incremental: full reload phù hợp dimension nhỏ, reference data, hoặc khi logic lịch sử thay đổi. Incremental phù hợp fact table lớn, event/order/log data, và pipeline chạy thường xuyên.

Date range parameterization: --start-date và --end-date giới hạn scan, write và downstream recompute. Đây là cách giảm cost rõ nhất khi warehouse tính tiền theo data scanned/compute time.

Parallel execution: Bruin chạy theo asset graph; các asset độc lập có thể chạy song song tùy cấu hình/runtime. Khi tăng concurrency, kiểm tra giới hạn source API, warehouse slots và lock table để tránh tự tạo bottleneck.

Asset caching: Không nên thiết kế dựa trên giả định Bruin cache intermediate results như build cache. Production cost control nên dựa vào materialization đúng, partition/date filter, incremental merge và warehouse tuning.

● ● ●cost-aware-run.sh

G 6. SECURITY & GOVERNANCE: Rủi ro trong setup hiện tại ▼

Credentials: .bruin.yml có thể chứa database path, token, service account hoặc custom secrets nên phải nằm trong .gitignore. Commit connection name và pipeline logic; không commit secret value.

Environment separation: dev/staging/prod cần connection isolation. Default environment nên là dev để tránh chạy nhầm production; production phải yêu cầu flag rõ ràng hoặc chạy qua CI/Cloud.

Bruin Cloud: Cloud quản lý credentials trong connection UI, validate connection, chạy schedule, lưu logs và hỗ trợ access control theo team. Git repo chỉ chứa project code.

Bruin MCP: MCP cho AI agent gọi Bruin commands và đọc context pipeline. AI không nên được đưa raw credentials trong prompt; rủi ro chính là agent có thể chạy query/command qua connection mà user đã cấp quyền. Giới hạn quyền connection, dùng read-only khi review, và tách môi trường.

P Phần 1 — Bruin Pro patterns ▼

Full selector syntax: dùng --asset để chạy một asset, --downstream để chạy các consumer bị ảnh hưởng, --upstream để build đủ dependencies trước một asset. Chạy toàn pipeline bằng đường dẫn pipeline.yml; trong repo nhiều pipeline, chọn đúng file pipeline thay vì chạy nhầm project root.

Date range execution: backfill lịch sử nên chia theo tháng/ngày để dễ retry và kiểm soát cost. Workflow tốt là validate, chạy một interval nhỏ, kiểm tra row count/quality, rồi lặp qua các interval lớn hơn.

bruin validate trước deployment: đưa vào CI/CD để bắt syntax lỗi, dependency lỗi, connection name thiếu và circular lineage trước khi merge/deploy.

bruin lineage để debug: đọc upstream để tìm input gây lỗi; đọc downstream để biết dashboard/table nào cần rerun hoặc cảnh báo consumer.

M Phần 2 — Bruin MCP Integration ▼

MCP là gì? MCP là Model Context Protocol. Trong context Bruin, MCP cho AI agent trong Cursor, VS Code, Claude Code... giao tiếp với Bruin để đọc project, chạy commands, truy vấn data, troubleshoot và sinh asset/config.

C Phần 3 — Bruin Cloud ▼

Bruin Cloud vs CLI: CLI phù hợp local development, validation, ad-hoc run và learning. Cloud dùng cùng project code nhưng thêm managed execution, UI logs, scheduling, retry, alerting, managed connections, lineage/metadata và collaboration.

Managed execution: thay vì cron/self-hosted runner, Cloud enable pipeline theo schedule, tạo run cho interval, lưu trạng thái từng asset và hiển thị quality check results.

Environment management: connection names trong code giữ ổn định; credential thật nằm trong Cloud connection UI. Dev/staging/prod tách bằng environment và quyền truy cập.

Team collaboration: Cloud phù hợp khi nhiều người cần xem run, lineage, logs, audit trail, access control và cùng review pipeline state mà không SSH vào server.

Khi nào upgrade: khi pipeline có SLA, cần schedule reliable, cần alert khi fail/stale, cần quản lý credentials tập trung, hoặc team không còn chấp nhận vận hành bằng terminal local.

O Phần 5 — Khi nào outgrow setup hiện tại ▼

Chuyển từ DuckDB local sang production DW

  • Data không còn vừa RAM/local disk, hoặc query scan mất quá lâu.
  • Nhiều consumer cần query cùng lúc.
  • Cần IAM, audit logs, backup, retention, partitioning và warehouse-level monitoring.

Thêm alerting/monitoring ngoài Bruin logs

  • Pipeline có SLA và fail lúc 2h sáng phải có người biết.
  • Data stale/row count anomaly quan trọng hơn trạng thái command success.
  • Cần dashboard vận hành theo latency, freshness, cost và quality trend.

Cần orchestrator ngoài Bruin built-in scheduling

  • Workflow phụ thuộc nhiều hệ thống ngoài data warehouse: API jobs, ML training, file transfer, business approvals.
  • Cần sensor/event-driven orchestration hoặc fan-out/fan-in phức tạp.
  • Đã có enterprise orchestrator và Bruin chỉ nên là asset execution layer.

Split thành multiple Bruin projects

  • Ownership khác nhau: finance, growth, product analytics có release cycle riêng.
  • Pipeline graph quá lớn, validate/run/review chậm và khó phân quyền.
  • Connection/security boundary khác nhau giữa domain hoặc environment.
Rule of thumb: đừng split chỉ vì folder dài. Split khi ownership, deployment boundary, security boundary hoặc SLA thật sự khác nhau.

D Debugging checklist theo thứ tự ưu tiên ▼
  1. Đọc error message đầu tiên, không chỉ dòng cuối cùng của stack trace.
  2. Xác định scope: fail ở connection, parse YAML, validation, run asset, quality check hay downstream query.
  3. Chạy bruin validate ./pipeline/pipeline.yml trước khi rerun để loại lỗi config/DAG.
  4. Dùng bruin lineage ./pipeline/pipeline.yml --asset <asset> để xem upstream/downstream.
  5. Rerun asset nhỏ nhất có thể bằng --asset và đúng --start-date/--end-date.
  6. Kiểm tra output bằng bruin query: row count, max timestamp, duplicate key, null key.
  7. Nếu có partial write hoặc duplicate, cleanup/rebuild interval trước khi chạy downstream.
  8. Chỉ dùng --full-refresh khi biết rõ blast radius và đã xác định downstream cần rebuild.
Logbook: Sau mỗi lỗi, mở logbook và ghi lại: command đã chạy, error gốc, root cause, fix cuối cùng, và cách phòng tránh lần sau.
S7 · Interview Prep
Interview Prep: từ Bruin project sang câu trả lời phỏng vấn
Mỗi câu có gợi ý trả lời và phần red flag để tránh trả lời hời hợt hoặc sai hướng.
J Junior questions Junior ▼
J1Data Platform là gì? Khác Data Warehouse và Data Pipeline thế nào?

Data Platform là lớp hệ thống giúp team build, chạy, kiểm soát và quan sát data workflows: ingestion, transformation, orchestration, quality, lineage, metadata, security. Data Warehouse là nơi lưu và query dữ liệu đã được tổ chức. Data Pipeline là một flow cụ thể đưa data từ source sang destination. Nói ngắn: warehouse là storage/compute, pipeline là workflow, platform là môi trường vận hành nhiều pipeline có governance.

Red flag: đừng nói Data Platform chỉ là BigQuery/Snowflake hoặc chỉ là một ETL script.
J2Asset, Pipeline, Connection trong Bruin là gì? Quan hệ giữa chúng thế nào?

Connection là cấu hình để Bruin truy cập source/destination như DuckDB, BigQuery, Postgres. Asset là unit công việc tạo/cập nhật một dataset: ingestr, SQL, Python, seed. Pipeline là nhóm assets có schedule, variables và default connections. Pipeline chạy asset graph theo dependency; asset dùng connection để đọc/ghi dữ liệu.

Red flag: đừng gọi asset là "một task bất kỳ" mà bỏ qua output/data contract của nó.
J3Tại sao khai báo dependency trong asset quan trọng hơn chạy script theo thứ tự thủ công?

Dependency biến thứ tự chạy thành metadata có thể validate, visualize và retry. Khi asset fail, Bruin biết downstream nào bị ảnh hưởng; khi một upstream đổi, team biết blast radius. Chạy script thủ công phụ thuộc trí nhớ người vận hành, khó audit và dễ chạy thiếu bước.

Red flag: đừng trả lời chỉ là "cho tiện"; điểm chính là reproducibility, lineage và incident recovery.
J4SQL asset khác Python asset thế nào, khi nào dùng cái nào?

SQL asset phù hợp transform set-based trong warehouse: join, filter, aggregate, dedup. Python asset phù hợp ingestion API/SDK, parsing file, logic procedural, ML hoặc validation tùy biến. Production rule: nếu data lớn và logic express được bằng SQL, pushdown vào warehouse; nếu cần gọi hệ thống ngoài SQL, dùng Python.

Red flag: đừng nói Python luôn mạnh hơn SQL; với warehouse transform lớn, Python local có thể là bottleneck.
M Mid-level questions Mid ▼
M5Variables start_date và end_date giải quyết vấn đề gì trong production pipeline?

Chúng biến pipeline thành interval-based execution. Cùng một asset có thể chạy daily, rerun một ngày lỗi, backfill một tháng hoặc giới hạn scan để giảm cost. Date range cũng là nền tảng cho incremental và idempotency vì mỗi run có boundary rõ.

Red flag: đừng nói chỉ để "lọc dữ liệu"; phải nhắc retry, backfill, cost và repeatability.
M6Incremental asset hoạt động thế nào? Chạy 2 lần cùng date range có duplicate không?

Incremental asset chỉ xử lý phần data mới hoặc một time interval thay vì rebuild toàn bộ. Có duplicate hay không phụ thuộc strategy: append sẽ duplicate nếu rerun cùng interval; merge/upsert với unique key hoặc time_interval delete-and-insert đúng window thì idempotent hơn. Luôn verify bằng row count và duplicate key check.

Red flag: đừng khẳng định incremental tự động luôn chống duplicate.
M7bruin validate vs bruin run — khi nào dùng cái nào, validate check những gì?

bruin validate dùng trước khi chạy/deploy để kiểm tra asset definitions, dependency graph, circular dependency, broken references và connection config. bruin run tạo execution thật và materialize output. CI nên validate trên PR; run dùng cho local test, scheduled execution hoặc deploy/backfill.

Red flag: đừng dùng run như cách đầu tiên để phát hiện lỗi syntax/config.
M8Làm sao debug khi pipeline fail mà log không đủ thông tin để biết asset nào gây ra?

Thu hẹp scope bằng lineage và selector. Chạy validate để loại lỗi graph/config. Dùng bruin lineage --asset quanh vùng nghi ngờ, rerun từng branch với date range nhỏ, query output upstream để xem schema/row count/freshness. Nếu ở Cloud, xem asset-level logs và quality check results thay vì chỉ pipeline-level status.

Red flag: đừng chỉ nói "xem log kỹ hơn"; phải có chiến thuật isolate failure.
S Senior questions Senior ▼
S9Pipeline ingest 50GB/ngày fail lúc 3h sáng sau 4 tiếng chạy, incremental asset ở dirty state — xử lý thế nào mà không mất data và không reprocess toàn bộ lịch sử?

Đầu tiên freeze downstream/alert consumer vì output có thể bẩn. Xác định failed asset, date interval và write strategy. Kiểm tra raw/source of truth còn nguyên, query partition bị ảnh hưởng: row count, duplicate key, max timestamp. Nếu dirty state nằm trong một interval, cleanup hoặc full refresh đúng interval/asset đó, rồi rerun --asset failed_asset --downstream --start-date ... --end-date .... Không full refresh toàn lịch sử trừ khi key/logic lịch sử sai. Sau incident, thêm idempotent merge/time_interval, quality check duplicate và runbook retry.

Red flag: đừng nói "rerun toàn bộ pipeline" ngay lập tức; đó là cost và risk lớn nếu 50GB/ngày.
S10Team 5 DE cùng develop trên Bruin project, branching và deployment strategy thế nào để tránh conflict và đảm bảo production không bị ảnh hưởng bởi dev changes?

Dùng branch per feature, PR review bắt buộc, CI chạy bruin validate và lineage diff trước merge. Tách environment dev/staging/prod bằng connection config; default là dev. Staging chạy sample/backfill nhỏ trước khi promote. Production deploy chỉ từ main/tag, credentials nằm trong Cloud/secret manager, không trong git. Với domain lớn, chia folder ownership hoặc split project khi release boundary khác nhau.

Red flag: đừng nói "ai cũng commit lên main rồi chạy prod nếu cần".
S11Thiết kế data platform cho e-commerce: 3 sources (DB, API, events), cần real-time dashboard và daily batch reports — chọn asset types, connection strategy, execution schedule thế nào?

DB orders/customers dùng ingestr hoặc CDC connector vào warehouse raw/staging theo incremental key. API catalog/payment dùng Python hoặc ingestr nếu connector có sẵn, với rate-limit/retry. Events realtime cần streaming path riêng cho dashboard low-latency; Bruin batch có thể consume curated event table để daily reports. Connections tách dev/staging/prod, service accounts least privilege. Schedule: realtime path near-real-time/event-driven; Bruin daily/hourly cho batch marts, quality checks và reconciliation giữa events/orders.

Red flag: đừng ép Bruin batch giải quyết toàn bộ real-time streaming nếu latency yêu cầu tính bằng giây.
S12Bruin MCP integration: AI generate asset code — review checklist những gì trước khi merge vào production?

Review asset name, grain, dependencies, connection/environment, date filters, materialization strategy, idempotency, unique key, quality checks, schema contract, secrets leakage, cost of query, and blast radius. Bắt AI hoặc CI chạy bruin validate, render SQL, query sample, duplicate/null checks. Business logic phải do owner xác nhận, không merge chỉ vì AI code chạy được.

Red flag: đừng nói "AI generate xong validate pass là merge"; validate không chứng minh business logic đúng.
