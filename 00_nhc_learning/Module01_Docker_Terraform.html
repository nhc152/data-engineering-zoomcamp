Docker giải quyết gì?

"Works on my machine" syndrome: cùng một script ingest data (ví dụ ingest_data.py trong 01-docker-terraform/docker-sql/pipeline/) chạy tốt trên máy A nhưng fail trên máy B vì khác version Python, package, OS dependency.

⚠️ Cảnh báo: Nếu chỉ dùng local virtualenv, bạn vẫn dễ gặp khác biệt system-level (libpq, timezone, locale, network) giữa dev machines.

Terraform giải quyết gì?

Click tay trên cloud console không reproducible: tạo GCS bucket / BigQuery dataset bằng UI thì nhanh lúc đầu, nhưng khó audit, khó review, khó recreate cho môi trường khác, và rất dễ drift cấu hình.

📌 Lưu ý: Trong repo, phần terraform/terraform/ đã hướng dẫn flow chuẩn init → plan → apply để biến hạ tầng thành mã version-controlled.

Trước Docker người ta làm gì?

  • Cài trực tiếp PostgreSQL + Python dependencies lên máy host.
  • Gửi checklist setup dài qua README/Wiki rồi hy vọng mọi người cài giống nhau.
  • Dùng VM nặng (VirtualBox/Vagrant) để cố gắng đồng bộ môi trường.

Trước Terraform người ta làm gì?

  • ClickOps: tạo resource thủ công trên AWS/GCP/Azure console.
  • Viết shell script gọi cloud CLI rời rạc, thiếu dependency graph rõ ràng.
  • Copy cấu hình giữa dev/staging/prod bằng tay → sai sót khó tránh.

🐳 Docker — Beginner → Intermediate → Senior ▼

Beginner level

Container là gì? Container là runtime package chứa app + dependency + config cần thiết để chạy nhất quán trên mọi môi trường.

Image là gì? Image là template bất biến để tạo container. Container là instance đang chạy của image.

📌 Analogy: Image giống "thiết kế chuẩn" của shipping container; container chạy thật là từng "thùng hàng" cụ thể đang di chuyển.

Intermediate level

Dockerfile layers: Mỗi instruction (FROM, RUN, COPY...) tạo layer. Build cache reuse layer cũ nếu instruction + context không đổi.

💡 Tip: Layer ordering optimization: đặt bước ít thay đổi (install system deps) lên trên, bước đổi thường xuyên (COPY source code) xuống dưới để tận dụng cache tối đa.

Docker Compose: Khi pipeline có nhiều service (PostgreSQL, pgAdmin, ingestion app), Compose gom toàn bộ dependency vào một file docker-compose.yaml, giúp up/down stack đồng bộ.

Senior level

  • Network modes: bridge (default local multi-container), host (hiệu năng cao hơn nhưng giảm isolation), overlay (multi-host/swarm).
  • Volume types: named volume (quản lý bởi Docker), bind mount (map thư mục host), tmpfs (in-memory, ephemeral).
  • Resource limits: giới hạn CPU/RAM để tránh một container ăn hết tài nguyên host và làm sập service khác.
🔥 Nguy hiểm: Không đặt memory/cpu limits trong môi trường share host có thể gây OOM kill ngẫu nhiên, đặc biệt khi ingest batch lớn.
Mental model: Docker = môi trường đóng gói chạy giống hệt nhau ở mọi nơi.

🏗️ Terraform — Beginner → Intermediate → Senior ▼

Beginner level

IaC (Infrastructure as Code) là gì? IaC là cách khai báo hạ tầng bằng code để version control, review, và recreate nhất quán.

Terraform vs ClickOps: ClickOps nhanh cho demo, nhưng khó audit và khó lặp lại; Terraform chậm hơn ban đầu nhưng scale team tốt hơn.

  • Provider: plugin để Terraform nói chuyện với cloud API (vd: google).
  • Resource: object hạ tầng cụ thể (bucket, dataset, VM...).
  • Variable: tham số hóa config cho nhiều môi trường.
  • Output: giá trị xuất ra sau apply để reuse ở bước khác.
💡 Best practice local dev: ưu tiên ADC bằng gcloud auth application-default login để Terraform lấy credentials tạm thời thay vì giữ Service Account Key JSON tĩnh trên máy.
● ● ●gcp-adc-login.sh
gcloud auth application-default login

Intermediate level

State file: Terraform lưu snapshot tài nguyên thật vào state để biết cần tạo/sửa/xóa gì lần chạy sau.

🔥 Nguy hiểm: Mất hoặc corrupt state có thể khiến Terraform "mù trạng thái", dẫn tới create trùng hoặc destroy nhầm tài nguyên.

Plan → Apply workflow: plan là bước preview diff hạ tầng; apply mới thực thi. Review plan trước apply giúp bắt sai sót sớm (name, region, quota, cost).

Senior level

  • Remote state (GCS backend): lưu state tập trung cho team thay vì local file rời rạc.
  • State locking: tránh 2 người apply đồng thời gây race condition trạng thái.
  • Workspace: tách state theo môi trường (dev/staging/prod) với cùng codebase.
💡 Tip: Trong CI/CD, pattern an toàn là plan trên Pull Request, chỉ apply sau khi merge và có approval.
Mental model: Terraform = git cho hạ tầng cloud.

Giải thích từng block + WHY

  • services: định nghĩa các thành phần runtime; repo tách pgdatabase và pgadmin để DB và UI quản trị chạy độc lập.
  • environment: inject credentials khởi tạo service; dùng env để tránh hardcode vào image layer, giúp thay theo môi trường dễ hơn.
  • volumes: giữ state lâu dài của DB/pgAdmin; nếu không có volume, down/up sẽ mất data hoặc mất cấu hình server trong pgAdmin.
  • ports: mở cổng từ host vào container để local tools/browser truy cập được.

Ports mapping: vì sao 5432:5432, 8080:8080?

Trong repo hiện tại, mapping thực tế là 5432:5432 và 8085:80:

  • 5432:5432: giữ đúng cổng mặc định PostgreSQL cả host và container, giúp lệnh CLI/driver quen thuộc.
  • 8085:80: pgAdmin trong container nghe ở cổng 80, nhưng host đổi sang 8085 để tránh xung đột với các web service khác.
⚠️ Cảnh báo: Nếu bạn đổi sang 8080:8080, pgAdmin phải thực sự listen trên 8080 trong container; với image này mặc định là 80.
📌 Tại sao compose hiện tại chưa có depends_on: đây là điểm yếu thực tế của file gốc — PostgreSQL thường mất 2–5 giây để ready; nếu ingest chạy ngay thì dễ gặp connection refused. Cách fix chuẩn là kết hợp healthcheck + depends_on.condition: service_healthy; chỉ dùng depends_on theo tên service thì mới đảm bảo container started, chưa đảm bảo DB ready.

Depends_on và thứ tự start

File compose hiện tại chưa khai báo depends_on. Nghĩa là Compose tạo service mà không enforce strict startup ordering. Về thực tế:

  • pgAdmin cần PostgreSQL đang reachable để connect thành công.
  • ingest container (chạy rời) cần DB sẵn sàng trước khi push data.
● ● ●docker-compose.yaml (depends_on + healthcheck example)
services:
  pgdatabase:
    image: postgres:18
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U root -d ny_taxi"]
      interval: 10s
      timeout: 5s
      retries: 5

  pgadmin:
    image: dpage/pgadmin4
    depends_on:
      pgdatabase:
        condition: service_healthy
💡 Tip: Pattern này làm startup ordering rõ ràng hơn: pgAdmin chỉ start khi DB đã healthy.

@click.option('--pg-user', default='root')
@click.option('--pg-pass', default='root')
@click.option('--pg-host', default='localhost')
@click.option('--pg-port', default=5432, type=int)
@click.option('--pg-db', default='ny_taxi')
@click.option('--year', default=2021, type=int)
@click.option('--month', default=1, type=int)
@click.option('--target-table', default='yellow_taxi_data')
@click.option('--chunksize', default=100000, type=int)
def run(pg_user, pg_pass, pg_host, pg_port, pg_db, year, month, target_table, chunksize):
    prefix = 'https://github.com/DataTalksClub/nyc-tlc-data/releases/download/yellow'
    url = f'{prefix}/yellow_tripdata_{year}-{month:02d}.csv.gz'

    engine = create_engine(
      f'postgresql+psycopg://{pg_user}:{pg_pass}@{pg_host}:{pg_port}/{pg_db}'
    )

    df_iter = pd.read_csv(
        url,
        dtype=dtype,
        parse_dates=parse_dates,
        iterator=True,
        chunksize=chunksize,
    )

    first = True
    for df_chunk in tqdm(df_iter):
        if first:
            df_chunk.head(0).to_sql(name=target_table, con=engine, if_exists='replace')
            first = False

        df_chunk.to_sql(name=target_table, con=engine, if_exists='append')

YAML liên quan trực tiếp tới script này là docker-compose.yaml (service name, port, network). Khi chạy ingestion container ngoài compose, script nhận params CLI để map vào infra runtime.

HCL không gọi trực tiếp ingest_data.py, nhưng tạo hạ tầng cloud upstream (bucket/dataset) cho các bước ingest và warehouse ở module sau.

Data flow trong script

  1. Build URL dataset theo year/month.
  2. Tạo SQLAlchemy engine tới PostgreSQL.
  3. Đọc CSV nén theo iterator + chunksize.
  4. Chunk đầu tạo schema table (head(0) + replace).
  5. Các chunk sau append dữ liệu liên tục.

Vì sao dùng chunks?

  • File taxi CSV lớn, đọc full dễ ngốn RAM.
  • Chunking giảm peak memory và giúp stream data từng phần vào DB.
  • Dễ quan sát tiến độ qua tqdm.

Connection string PostgreSQL — từng tham số

  • postgresql+psycopg://: SQLAlchemy dialect + driver.
  • {pg_user}:{pg_pass}: credentials DB.
  • @{pg_host}:{pg_port}: endpoint DB (localhost hoặc service name trong Docker network).
  • /{pg_db}: database đích (ny_taxi).

Điểm yếu hiện tại của script

  • Chưa idempotent tuyệt đối: mỗi lần chạy với cùng table sẽ replace rồi append lại toàn bộ.
  • Error handling tối thiểu: chưa bắt lỗi network/DB/auth/timeouts theo từng class lỗi.
  • Không có checkpoint/resume: fail giữa chừng phải ingest lại từ đầu.
  • Chưa có validation dữ liệu: chưa check schema drift/null anomalies trước khi load.

Chi tiết kỹ thuật quan trọng từ repo

  • Vì sao dùng Int64 thay vì int64: dataset thực tế có missing values; int64 thường không chứa được NaN, còn Int64 (pandas nullable integer) thì giữ được null mà vẫn bảo toàn semantic kiểu số nguyên.
  • uv sync --locked khác pip install: cờ --locked buộc dùng đúng versions trong uv.lock, giúp tránh dependency drift giữa local/CI/container.
  • pgAdmin map 8085:80 (không phải 8080:80): service trong container listen cổng 80, còn host chọn 8085 để giảm xung đột với các web app/dev server thường chiếm 8080.

💥 Phần 5 — Cố tình phá (Break) & Debug từng bước ▼

Scenario 1 — Đổi sai port PostgreSQL trong compose

  • Phá: đổi mapping thành 5434:5432 nhưng app vẫn connect 5432.
  • Lỗi thường gặp: connection timeout / refused khi dùng host port cũ.
  • Debug: chạy docker ps kiểm tra port thực tế; chạy docker-compose logs pgdatabase; đối chiếu lại tham số --pg-port.

Scenario 2 — Xóa volume khi container đang chạy

  • Phá: cố remove volume DB đang mount.
  • Kết quả: Docker thường chặn xóa nếu volume đang in-use; nếu stop rồi xóa volume thì data mất.
  • Debug: docker volume ls + docker volume inspect <name>; xác nhận container nào đang giữ volume.

Scenario 3 — Sai password DB trong env

  • Phá: sửa POSTGRES_PASSWORD hoặc --pg-pass không khớp.
  • Lỗi thường gặp: password authentication failed (không phải connection refused trong phần lớn trường hợp).
  • Tìm log: docker-compose logs pgdatabase, log container ingest, và traceback SQLAlchemy trong terminal.

Scenario 4 — Xóa terraform.tfstate

  • Phá: xóa file state local rồi chạy terraform apply lại.
  • Chuyện xảy ra: Terraform mất mapping resource đã tạo; có thể đề xuất tạo lại resource hoặc gặp conflict name đã tồn tại.
  • Debug: kiểm tra backup state, dùng terraform import khi cần reclaim resource, tránh apply mù.

Scenario 5 — Mô phỏng Partial Insert (đang chạy thì Ctrl+C)

  • Phá: chạy ingest và ấn Ctrl+C giữa chừng khi progress bar chưa hoàn tất.
  • Chuyện xảy ra: bảng bị nạp một phần dữ liệu vì run dừng giữa loop append.
  • Bản chất kỹ thuật: chunk đầu dùng if_exists='replace' nên lần chạy lại sẽ drop/recreate table rồi nạp lại từ đầu — không tạo duplicate do pipeline, nhưng tốn lại toàn bộ compute/time cho full reload.
  • Kiểm tra: đếm dòng trước khi ngắt và sau khi chạy lại full:
    ● ● ●verify-partial-insert.sql
    SELECT COUNT(*) FROM yellow_taxi_data;

Scenario 6 — Chạy ingest với --pg-host=localhost

  • Phá: chạy docker run ingest nhưng truyền --pg-host=localhost thay vì --pg-host=pgdatabase.
  • Lỗi thường gặp: connection refused hoặc could not connect to server.
  • Tại sao: mỗi container có network namespace riêng; localhost trong container trỏ về chính container đó, không trỏ về host machine hay container khác.
  • Fix: luôn dùng container/service name làm hostname khi giao tiếp container-to-container (ở lab này là pgdatabase).
  • Lưu ý: đây là lỗi phổ biến nhất của người mới học Docker networking.
⚠️ Roadmap alignment (M01 D5–D7): bắt buộc thử các lỗi kiểu port/env/state và ghi lại logbook theo mẫu Break → Root cause → Fix → Lesson learned.

🔍 S4 — Production Thinking (Senior Questions)

Ở production, Docker + Terraform không được đánh giá bằng việc “chạy được”, mà bằng khả năng recover nhanh, scale đúng chỗ, và kiểm soát cost + security. 6 câu hỏi dưới đây là khung tư duy Senior bạn cần luyện từ Module 01.

1) Failure — Nếu pipeline fail lúc 3h sáng, debug từ đâu?

WHY: Lúc incident, sai lầm phổ biến nhất là sửa ngay mà chưa khoanh vùng. Module này có 2 lớp failure chính: container runtime (Docker) và infra drift/config (Terraform).

● ● ●incident-triage.sh
docker compose ps
docker logs pgdatabase --tail 120
docker logs pgadmin --tail 120

docker exec -it pgdatabase psql -U root -d ny_taxi -c "SELECT COUNT(*) FROM yellow_taxi_data;"

terraform state list
terraform plan
  • Bước 1: Xác nhận service nào down/restart loop.
  • Bước 2: Đọc logs Postgres trước, rồi ingest step.
  • Bước 3: Chỉ chạy lại ingest sau khi DB healthy.
  • Bước 4: Nếu cloud path lỗi, check state drift trước apply.
⚠️ Cảnh báo: Re-run mù khi DB chưa sẵn sàng sẽ tạo chuỗi lỗi giả, làm incident kéo dài.
2) Scale — Nếu volume tăng x100, bottleneck ở đâu?

WHY: Script ingest_data.py đọc CSV theo chunk và ghi vào Postgres local. Mô hình này tốt để học, nhưng bottleneck sẽ xuất hiện nhanh khi volume tăng mạnh.

  • Single Postgres container: write throughput hạn chế.
  • CSV parse bằng pandas: CPU + memory pressure.
  • Network + disk I/O local volume: nghẽn khi ghi liên tục.

Hướng scale:

  • Near-term: tune chunksize, batch insert, tách download và load.
  • Mid-term: dùng object storage làm staging.
  • Senior path: chuyển sang columnar format + distributed compute cho transform lớn.
💡 Tip: Scale đúng là loại bỏ bottleneck lớn nhất trước, không phải tăng đồng loạt CPU/RAM.
3) Idempotency — Chạy lại 2 lần có safe không?

WHY: Batch pipeline production phải re-run được sau lỗi mà không để lại duplicate hoặc state nửa vời.

● ● ●ingest-idempotency.py
# chunk đầu thay table
c = next(df_iter)
c.to_sql(name=table_name, con=engine, if_exists='replace')

# các chunk sau append
for c in df_iter:
    c.to_sql(name=table_name, con=engine, if_exists='append')

Pattern này cho full refresh là tương đối an toàn vì mỗi run reset table về trạng thái chuẩn. Trade-off: table lớn sẽ tốn thời gian; production thường đi theo incremental + dedup key.

📌 Lưu ý: Idempotency là thuộc tính của outcome cuối, không chỉ của một câu lệnh SQL.
4) Monitoring — Metric nào cho biết pipeline healthy?

WHY: Không đo thì không vận hành được. Với module này, hãy luyện monitoring mindset ngay cả khi chạy local.

  • Freshness: thời điểm ingest thành công gần nhất.
  • Volume: số rows nạp mỗi run, so với baseline.
  • Duration: thời gian từ start đến loaded.
  • Error rate: tỷ lệ run fail.
  • Container health: restart count, disk usage volume.
● ● ●freshness-check.sql
SELECT
  MAX(tpep_pickup_datetime) AS latest_pickup,
  COUNT(*) AS total_rows
FROM yellow_taxi_data;
5) Cost — Ước lượng và tối ưu chi phí thế nào?

WHY: Terraform làm provision nhanh, nhưng cũng có thể tạo chi phí sai rất nhanh nếu apply thiếu kiểm soát.

  • GCS Standard Storage: khoảng $0.02/GB/tháng (tham chiếu mức phổ biến cho bài lab).
  • BigQuery analysis: khoảng $5/TB data processed, nên query scan dư thừa sẽ đội chi phí rất nhanh.
  • Egress phát sinh khi region mismatch.
  • Orphan resources do state drift làm “rò” chi phí dài hạn.
● ● ●cost-estimation.sh
infracost breakdown --path .

Practice chuẩn: chạy infracost + review terraform plan trước apply, gắn label cost owner, và teardown tài nguyên học tập khi không dùng.

⚠️ Cảnh báo: Để lâu môi trường lab không destroy là lỗi cost phổ biến nhất của người mới làm cloud.
6) Security — Credential đã xử lý đúng chuẩn chưa?

WHY: Nhiều incident bắt đầu từ hardcoded secrets hơn là từ bug logic.

  • Không hardcode password/key trong source.
  • Tách secrets khỏi repo (Secret Manager/Vault).
  • Áp dụng least privilege cho service account.
  • Không commit state file nếu chứa metadata nhạy cảm.
  • Quét secrets trước commit.
● ● ●.gitignore (security baseline)
*.env
terraform.tfstate
terraform.tfstate.backup
*.tfvars
  • Container chạy root mặc định — rủi ro: nếu app có lỗ hổng, attacker có thể chiếm quyền root trong container và mở rộng blast radius.
● ● ●Dockerfile (run as non-root)
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser
  • Service Account Key JSON trên máy cá nhân — rủi ro: nếu key bị leak (commit nhầm git, máy bị compromise), attacker có thể dùng key đó truy cập tài nguyên GCP theo quyền đã cấp.
  • Alternative production tốt hơn: Workload Identity Federation cấp credentials tạm thời cho workload (GitHub Actions/VM) mà không cần lưu key file lâu dài.
  • Nguyên tắc: credentials không nên tồn tại dưới dạng file tĩnh trên disk nếu có thể tránh được.
🔥 Nguy hiểm: Một lần push nhầm credential public có thể dẫn đến incident bảo mật và chi phí cloud ngoài kiểm soát.

🚀 S5 — Kiến thức nâng cao: Từ Zoomcamp lên Production

S5 tập trung vào phần roadmap gọi là Pro-tier patterns: các kỹ thuật chưa bắt buộc ở workshop nhưng rất quan trọng khi vận hành thật với team và CI/CD.

A) Docker Pro-tier Patterns

1) Multi-stage build để giảm image size

WHY: Image nhẹ pull nhanh hơn, scan security nhanh hơn, deploy ổn định hơn.

  • Builder stage: cài full dependencies build.
  • Runtime stage: chỉ giữ artifacts cần chạy.

2) Resource limits + health checks

WHY: Tránh 1 container “ăn sạch” tài nguyên máy và giúp orchestrator phát hiện unhealthy service.

  • Define CPU/memory limits cho service critical.
  • Health check DB/app để gate step ingest tiếp theo.

3) Immutable image tagging

WHY: Dùng tag theo commit SHA giúp rollback chính xác, tránh drift do :latest.

B) Terraform Pro-tier Patterns

1) Remote backend + state locking

WHY: Team nhiều người mà dùng local state sẽ rất dễ ghi đè và drift.

  • Remote backend (ví dụ GCS/S3) cho shared state.
  • State locking để tránh apply đồng thời.

2) Workspace strategy theo environment

WHY: Tách dev/staging/prod giúp kiểm soát blast radius khi apply thay đổi.

3) CI/CD: plan trước, apply sau approval

WHY: Review infra diff như review code, giảm rủi ro thay đổi ngoài ý muốn.

● ● ●terraform-ci-flow.sh
terraform fmt -check
terraform init
terraform validate
terraform plan -out=tfplan
# Manual approval
terraform apply tfplan

D) Khi nào outgrow stack hiện tại?

Dấu hiệu outgrow Docker Compose:

  • Cần autoscaling theo workload.
  • Cần rolling deploy, self-healing ở mức cluster.
  • Cần secret/config management tập trung cho nhiều service.

Hướng thay thế: Kubernetes (AKS/GKE/EKS) khi cần orchestration production-grade.

Dấu hiệu outgrow Terraform thuần module đơn giản:

  • Codebase IaC lớn, nhiều shared module phức tạp.
  • Nhu cầu policy-as-code và guardrail chặt.
  • Cần abstraction cao hơn cho app-centric infra.

Hướng thay thế/cộng thêm: Terragrunt cho quản trị module lớn; Pulumi/CDK khi team cần abstraction bằng ngôn ngữ lập trình.

📌 Lưu ý: Outgrow không có nghĩa tool cũ “dở”, mà là requirements mới đã vượt ngưỡng thiết kế ban đầu.
E) Senior mindset summary cho Module 01

  • Tối ưu cost ngay từ design: region, lifecycle, retention, teardown.
  • Xử lý secrets như tài sản nhạy cảm, không coi là config tiện tay.

Lỗi thường gặp — Docker & Terraform
Đọc đúng error message, khoanh vùng nguyên nhân, fix dứt điểm và phòng tránh tái diễn.
🧯 10 lỗi phổ biến trong Module 01 ▼
Lỗi (error message thực tế) Nguyên nhân Cách fix Cách phòng tránh
port is already allocated (docker-compose up) Port host (thường 5432 hoặc 8085) đã bị process/container khác chiếm. Kiểm tra docker ps/netstat, stop container/process đang chiếm port hoặc đổi port mapping. Chuẩn hóa port conventions cho team và ghi rõ trong README + compose.
connection refused (app connect DB) DB chưa ready, sai host/port, hoặc app dùng localhost thay vì service name trong compose network. Xác nhận DB healthy, dùng đúng host (pgdatabase) và port nội bộ. Thêm health check và startup ordering; luôn verify connectivity trước ingest.
No such file or directory (volume mount path sai) Bind mount trỏ đến path host không tồn tại hoặc sai relative path. Kiểm tra path tuyệt đối/tương đối, tạo đúng thư mục host trước khi mount. Dùng named volume khi không cần bind code trực tiếp, hạn chế path phụ thuộc máy cá nhân.
Container exits immediately (exit code 1) Entry command fail do thiếu env, dependency lỗi, hoặc script crash khi start. Đọc docker logs <container>, sửa command/env và rebuild nếu cần. Tách startup check rõ ràng, fail-fast với error message dễ đọc.
permission denied (file/folder trong container) User trong container không có quyền đọc/ghi volume hoặc file executable thiếu quyền. Chỉnh owner/permission đúng, kiểm tra user runtime và chmod/chown phù hợp. Thiết kế image với quyền tối thiểu nhưng đúng mục đích; tránh chạy root mặc định nếu không cần.
Data mất sau docker-compose down Không mount persistent volume hoặc dùng sai volume path cho DB data dir. Gắn named volume đúng thư mục dữ liệu PostgreSQL, không xóa volume khi teardown. Review compose trước khi chạy; phân biệt rõ down và down -v.
Error: No configuration files found (terraform init) Chạy lệnh sai thư mục, không có file .tf trong working directory. cd đúng vào folder Terraform (vd: terraform_with_variables) rồi chạy lại. Thiết lập shell prompt/alias hiển thị thư mục hiện tại trước khi chạy IaC.
Error acquiring the state lock (terraform apply) Một process khác đang giữ lock state hoặc lock cũ chưa được release. Xác định process đang chạy, chờ hoàn tất; chỉ force-unlock khi chắc chắn không còn apply nào active. Dùng workflow apply tuần tự qua CI/CD, tránh nhiều người apply tay cùng lúc.
403 Forbidden (terraform apply lên GCP) Service account/user thiếu IAM permission cho resource cần tạo. Đối chiếu role cần thiết (Storage, BigQuery...) và cấp đúng quyền tối thiểu. Chuẩn hóa IAM matrix theo môi trường, validate quyền trước khi apply.
terraform.tfstate bị corrupt hoặc mất State local hỏng/xóa nhầm, đồng bộ thủ công sai hoặc merge conflict state file. Khôi phục từ backup/remote backend; dùng terraform import để re-attach resource nếu cần. Dùng remote state + locking + backup policy; không chỉnh state bằng tay trừ tình huống recovery.
psycopg.OperationalError: connection to server at "localhost" failed Chạy ingest container với --pg-host=localhost; trong container, localhost trỏ về chính nó chứ không phải PostgreSQL container. Đổi host thành --pg-host=pgdatabase (tên container/service trên Docker network). Luôn dùng container name làm hostname khi giao tiếp nội bộ Docker network.

Debugging checklist theo thứ tự ưu tiên

  1. Đọc nguyên văn lỗi đầu tiên (đừng bắt đầu từ stack trace cuối cùng).
  2. Xác định lớp lỗi: runtime container, network, credentials, hay state/infra.
  3. Kiểm tra trạng thái service nền: DB/container/provider đã healthy chưa.
  4. Kiểm tra config đầu vào: host, port, path, env vars, IAM, terraform working dir.
  5. Sửa một biến mỗi lần và re-run để xác nhận đúng root cause.
  6. Chụp lại evidence (command + output) trước/sau fix để tránh "fix may mắn".
  7. Đánh giá phòng ngừa (health check, retry, idempotency, remote state, runbook).

S7 · Interview Prep
Interview Prep — Junior → Mid → Senior
Mỗi câu hỏi đều cần trả lời theo WHY + Trade-off + Failure handling, không chỉ định nghĩa.
🔵 Junior 1) Container khác VM ở điểm nào cơ bản nhất? ▼

Gợi ý trả lời: Container chia sẻ kernel của host nên nhẹ và start nhanh; VM ảo hóa cả OS nên isolation mạnh hơn nhưng tốn tài nguyên hơn. Container phù hợp packaging app/pipeline, VM phù hợp workload cần boundary hệ điều hành riêng.

🚩 Red flag: đừng nói... "Container là VM nhẹ nên giống hệt VM" — câu này bỏ qua khác biệt kiến trúc cốt lõi.
🔵 Junior 2) docker-compose up -d khác docker run ở chỗ nào? ▼

Gợi ý trả lời: docker run thường chạy 1 container theo câu lệnh đơn; docker-compose up -d chạy nhiều service theo declarative file (network, volume, env, dependency) và dựng stack đồng bộ.

🚩 Red flag: đừng nói... "Compose chỉ là viết tắt cho nhiều lệnh run" — thiếu phần orchestration config-as-code.
🔵 Junior 3) Terraform state file là gì và lưu ở đâu mặc định? ▼

Gợi ý trả lời: State là snapshot mapping giữa code Terraform và resource thực tế. Mặc định Terraform lưu local tại terraform.tfstate trong thư mục chạy lệnh; team production thường chuyển sang remote backend để tránh mất/corrupt và giảm conflict.

🚩 Red flag: đừng nói... "State chỉ là cache, xóa cũng không sao" — xóa state có thể làm Terraform mất toàn bộ nhận thức hiện trạng.
🔵 Junior 4) Tại sao phải chạy terraform plan trước terraform apply? ▼

Gợi ý trả lời: plan cho bạn preview chính xác resource nào sẽ add/change/destroy trước khi tác động môi trường thật; đây là bước kiểm soát rủi ro, chi phí và sai config. Trong team, plan cũng là artifact review trước approval apply.

🚩 Red flag: đừng nói... "Apply luôn cho nhanh" — bỏ bước plan là anti-pattern với hạ tầng production.
🟡 Mid 5) Làm sao đảm bảo data không mất khi container restart? ▼

Gợi ý trả lời: Tách state khỏi lifecycle container bằng persistent volume (named volume hoặc storage ngoài host), mount đúng data directory và kiểm soát backup policy. Restart policy chỉ giúp process sống lại, không thay thế persistence strategy.

🚩 Red flag: đừng nói... "Dùng restart: always là đủ" — service lên lại nhưng data vẫn có thể mất.
🟡 Mid 6) Nếu 2 developer cùng chạy terraform apply, chuyện gì xảy ra? ▼

Gợi ý trả lời: Có nguy cơ race condition lên state: một người giữ lock, người còn lại gặp lock error; tệ hơn là khi không lock chuẩn thì có thể gây drift hoặc ghi đè trạng thái. Quy trình đúng là remote state + locking + apply tập trung qua CI.

🚩 Red flag: đừng nói... "Không sao vì Terraform tự merge" — Terraform không phải git merge cho hạ tầng.
🟡 Mid 7) Tại sao nên dùng variables.tf thay vì hardcode trong main.tf? ▼

Gợi ý trả lời: Variables tách input khỏi logic giúp tái sử dụng cùng module cho nhiều môi trường, giảm copy-paste, giảm lỗi thao tác khi đổi project/region/name. Nó cũng giúp review diff tập trung vào logic hạ tầng thay vì noise cấu hình.

🚩 Red flag: đừng nói... "Chỉ để code đẹp hơn" — lợi ích chính là kiểm soát thay đổi và giảm rủi ro vận hành.
🟡 Mid 8) Docker image build lại mỗi lần deploy có vấn đề gì về performance? ▼

Gợi ý trả lời: Build full mỗi lần làm chậm pipeline CI/CD, tốn compute và network pull/push, nhất là khi dependencies lớn. Tối ưu bằng layer caching, multi-stage build, base image ổn định và chỉ rebuild phần thực sự thay đổi.

🚩 Red flag: đừng nói... "Build chậm không ảnh hưởng gì" — tốc độ deploy chậm trực tiếp giảm incident recovery speed.
🔴 Senior 9) Thiết kế Docker setup cho pipeline production với 10 services, đảm bảo không có single point of failure ▼

Gợi ý trả lời: Tách service theo trách nhiệm, mỗi service có health check + restart policy + resource limits; stateful components dùng storage bền vững và backup. Đặt load balancer trước service critical, tách observability stack riêng, và có cơ chế graceful degradation khi một service phụ trợ fail.

🚩 Red flag: đừng nói... "Chỉ cần docker-compose với 1 host mạnh" — 1 host là SPoF rõ ràng.
🔴 Senior 10) Terraform state ở local bị mất sau khi đã apply lên GCP — xử lý thế nào? ▼

Gợi ý trả lời: Dừng apply ngay để tránh drift thêm. Khôi phục từ backup nếu có; nếu không, tạo state mới cẩn trọng và terraform import các resource hiện hữu để tái lập mapping. Sau recovery, chuyển sang remote backend + locking để tránh lặp lại.

🚩 Red flag: đừng nói... "Cứ apply lại cho Terraform tự hiểu" — có thể tạo trùng resource hoặc phá naming constraints.
🔴 Senior 11) Ingest pipeline fail sau 6 tiếng, đã insert 80% data — restart toàn bộ hay tiếp tục? Design idempotent solution ▼

Gợi ý trả lời: Không restart mù từ đầu nếu tốn thời gian và dễ tạo inconsistency. Thiết kế checkpoint theo batch/window + key dedup/merge (UPSERT) để resume từ mốc cuối cùng an toàn. Idempotent outcome quan trọng hơn việc rerun “đơn giản”.

🚩 Red flag: đừng nói... "Xóa bảng rồi nạp lại luôn" trong mọi bối cảnh — cách này có thể phá SLA và làm mất khả năng truy vết.
🔴 Senior 12) Team có 5 DE cùng dùng chung GCP project với Terraform — quy trình làm việc tránh conflict thế nào? ▼

Gợi ý trả lời: Chuẩn hóa workflow: branch/PR bắt buộc, plan trên PR, apply sau merge qua CI với quyền tập trung; dùng remote state + locking + convention module rõ ràng theo domain. Tách môi trường bằng workspace hoặc project riêng để giảm blast radius.

🚩 Red flag: đừng nói... "Mỗi người tự apply local cẩn thận là được" — cách này không scale và thiếu auditability.
