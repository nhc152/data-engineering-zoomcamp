🎯 Analytics Engineering và dbt giải quyết vấn đề gì? ▼

Vấn đề Analytics Engineering giải quyết

"SQL script rải rác không ai biết chạy theo thứ tự nào, không có test, không có docs".

Khi team lớn dần, mỗi analyst/developer tạo một batch SQL riêng, chạy thủ công ở nhiều nơi. Kết quả là cùng một metric nhưng mỗi dashboard cho một con số khác nhau vì logic không được quản trị tập trung.

Vấn đề dbt giải quyết

"transformation ad-hoc, không reproducible, không version-controlled, không có lineage".

dbt đưa transformation vào cùng quy trình engineering: code review, dependency graph, testing, docs, CI/CD. Bạn không chỉ "chạy SQL", bạn xây data product có vòng đời rõ ràng.

🔥 Nguy hiểm: nếu không có lineage và test, sai logic ở staging có thể âm thầm đi tới mart/reporting nhiều tuần trước khi business phát hiện.

Trước dbt, người ta làm gì?

  • Stored procedures: gom logic trong warehouse nhưng khó review/version hóa theo chuẩn software.
  • Notebook pipelines: nhanh để thử nghiệm nhưng dễ drift môi trường, khó tái lập tuyệt đối.
  • Manual SQL: copy/paste query theo checklist hoặc cron script rời rạc, phụ thuộc con người.
⚠️ Cảnh báo: các cách cũ vẫn chạy được ở quy mô nhỏ, nhưng thường vỡ khi số model và số người cùng chạm vào data tăng nhanh.

Beginner — ELT vs ETL và vì sao cloud DW đổi paradigm

ETL cổ điển: transform nặng ở tầng compute trước khi load vào warehouse. ELT hiện đại: load raw trước, transform trong warehouse bằng SQL engine mạnh.

Analogy: "bếp nhà hàng vs bếp tự nấu". ETL cũ giống tự nấu tất cả ở nhà rồi mới mang ra bán; ELT giống đưa nguyên liệu vào bếp nhà hàng công suất lớn (BigQuery/Snowflake), chế biến tại chỗ theo order.

Beginner — Dimensional modeling trực quan

  • Fact table: chứa sự kiện đo lường (trip, order, payment), nhiều dòng, cập nhật liên tục.
  • Dimension table: chứa ngữ cảnh mô tả (zone, vendor, customer, product).
  • Star schema: fact ở giữa, dimensions xung quanh để query BI dễ hiểu và ổn định.

Intermediate — dbt là gì ở mức workflow

  • Transformation workflow: SQL files + dependency graph + run order tự động.
  • Jinja compilation: SQL được compile theo adapter, biến/macro/environment.
  • Materialization: quyết định output model là view/table/incremental/ephemeral.

Senior — dbt trong modern data stack

dbt nằm giữa ingestion và serving: raw data từ pipeline đi vào warehouse, dbt chuẩn hóa + mô hình hóa thành semantic layer để BI/ML/features dùng lại nhất quán. Đây là "control plane" cho quality + lineage của transformation layer.

💡 Mental model: dbt = git + CI/CD + docs + testing cho SQL transformation.

🧩 dbt Core Concepts (Beginner → Senior) ▼

Beginner — Models, Sources, Seeds là gì?

  • Models: file SQL trong models/, mỗi model ~ một SELECT thể hiện business logic.
  • Sources: khai báo bảng raw bên ngoài dbt trong sources.yml, dùng để track freshness + lineage upstream.
  • Seeds: CSV tĩnh version-controlled (ví dụ payment lookup), nạp vào warehouse để tránh hard-code CASE lặp lại.

Beginner — ref() vs source()

FunctionTrỏ tớiÝ nghĩa lineage

Intermediate — Macros + Jinja

Macros giúp tránh lặp SQL và chuẩn hóa logic cross-model. Jinja cho phép condition theo adapter/target/environment, ví dụ repo dùng macro tính trip duration và macro mapping vendor data.

💡 Tip: chỉ macro hóa logic có tính lặp hoặc cần abstraction cross-database; macro hóa quá sớm sẽ làm model khó đọc.

Senior — Incremental model pitfalls

  • unique_key: bắt buộc để merge/upsert đúng record thay đổi.
  • merge strategy: cần tương thích warehouse adapter và business semantics.
  • late-arriving data: nếu filter incremental quá hẹp, record đến muộn sẽ bị bỏ sót.
  • on_schema_change: phải định nghĩa rõ (append/ignore/fail) để tránh drift âm thầm.

Senior — dbt build vs dbt run

dbt run chỉ build models; dbt build chạy cả models + tests + snapshots + seeds theo graph. Trong production, dbt build là lệnh trọng tâm vì nó biến data quality thành gate thay vì hậu kiểm.

💡 Mental model: dbt model = 1 SELECT query → dbt lo phần còn lại.
⚠️ Warn: chạy mỗi dbt run trong prod có thể đẩy dữ liệu sai vào downstream nếu tests không được kích hoạt đồng bộ.
🔥 Danger: incremental sai key có thể nhân bản dữ liệu nhiều tháng trước khi ai đó phát hiện metric bị inflate.

Giải thích từng block

  • name/profile: định danh project và profile kết nối warehouse.
  • model-paths/seed-paths/macro-paths: nói cho dbt biết vị trí code assets.
  • vars: biến cấu hình dùng cho môi trường dev/sample run.

Database Error: relation does not exist ref() trỏ đến model chưa build hoặc model nằm ngoài selection hiện tại. Chạy model upstream trước: dbt run --select +model_name hoặc dbt build --select +model_name. Dùng CI để build graph theo dependency; tránh chạy rời từng model không có +.

Compilation Error: Expected NodeType.Model Dùng source() thay cho ref() khi gọi model trung gian. Đổi sang ref('model_name') cho inter-model dependency. Quy ước rõ: raw ingestion dùng source(), transform layers dùng ref().
Error: node was not found Typo tên model trong --select hoặc model bị đổi tên. Chạy dbt ls --select ... để xác thực selector trước khi run/build. Dùng selector theo path/tag thay vì hardcode tên lẻ khi có thể.

Runtime Error: out of memory DuckDB OOM khi build full dataset trên máy local RAM thấp. Chạy subset theo date, giảm columns, hoặc chuyển workload nặng sang BigQuery. Giữ local cho dev/prototype; production scale dùng warehouse managed.
Error acquiring state lock Hai process dbt chạy đồng thời cùng state/artifacts. Stop process trùng, xóa lock hợp lệ nếu job cũ đã chết, chạy lại tuần tự. Thiết kế orchestrator tránh overlap schedule; mỗi env chỉ một writer tại một thời điểm.
Package not found dbt deps fail do registry unreachable hoặc version mismatch. Kiểm tra mạng, pin version hợp lệ trong packages.yml, chạy lại dbt deps. Pin package versions rõ ràng và cache dependency trong CI.
env_var GCP_PROJECT_ID not set Thiếu biến môi trường khi chạy target BigQuery. Set env var trong shell/CI secret rồi chạy lại profile target. Document contract env vars bắt buộc cho từng environment.

compilation error in macro Jinja syntax lỗi trong macro custom (dấu ngoặc, filter, biến context). Chạy dbt compile, sửa syntax và test macro với input nhỏ. Review macro bằng PR checklist + unit test cho macro quan trọng.
🧭 Phần 2 — Debugging checklist theo thứ tự ưu tiên ▼
  1. Copy nguyên văn error message (không paraphrase) và xác định lỗi ở compile hay runtime.
  2. Thu hẹp phạm vi bằng selector: chạy model nhỏ nhất tái hiện lỗi với dbt build --select ....
  3. Kiểm tra dependency graph: xác thực ref(), source(), và upstream readiness.
  4. So khớp environment: profile target, env vars, credentials, package versions.
  5. Inspect data reality: duplicate/null/schema drift trực tiếp trong warehouse.
  6. Apply fix nhỏ nhất có thể, rồi chạy lại theo thứ tự compile → run/build → test.
  7. Xác nhận chống tái phát: thêm test, contract, freshness check hoặc CI gate tương ứng.
📝 Action bắt buộc: mở Logbook và ghi lại lỗi vừa gặp (symptom, root cause, fix, lesson learned) để build tư duy Senior qua từng incident.

S7 — Interview Prep (Junior → Mid → Senior)

🎯 Cách dùng section này: click từng câu để mở gợi ý trả lời + red flag ▼

Mục tiêu không phải học thuộc lòng, mà là trả lời theo cấu trúc: Context → Decision → Trade-off → Failure mode.

🔵 Junior 1) dbt model là gì? Khác stored procedure thế nào? ▶

Gợi ý trả lời: dbt model là file SQL dạng SELECT declarative, dbt compile + materialize thành view/table/incremental trong warehouse. Stored procedure là code procedural chạy theo control-flow; khó review lineage hơn và thường gắn chặt engine-specific syntax.

Red flag: đừng nói... “dbt chỉ là tool chạy SQL nên giống stored procedure.”

🔵 Junior 2) ref() và source() khác nhau thế nào, khi nào dùng? ▶

Gợi ý trả lời: source() trỏ dữ liệu raw có sẵn (ngoài dbt ownership), còn ref() trỏ model do dbt build và tự tạo dependency graph. Rule thực hành: raw → staging dùng source(); staging/intermediate/marts gọi nhau bằng ref().

Red flag: đừng nói... “Dùng cái nào cũng được, miễn query chạy.”

🔵 Junior 3) Tại sao staging layer nên dùng view thay vì table? ▶

Gợi ý trả lời: staging chủ yếu chuẩn hóa schema/typing nhẹ, nên view giảm storage duplication và luôn phản ánh raw mới nhất. Chỉ chuyển sang table khi view bị scan lặp gây cost/latency cao hoặc logic quá nặng.

Red flag: đừng nói... “View luôn tốt hơn table trong mọi trường hợp.”

🔵 Junior 4) dbt test chạy lúc nào trong pipeline, fail test có stop pipeline không? ▶

Gợi ý trả lời: khi dùng dbt build, test chạy sau khi model liên quan được materialize; test fail sẽ làm step/job fail để chặn downstream publish. Với dbt run đơn thuần thì không tự chạy test nếu không gọi thêm dbt test.

Red flag: đừng nói... “Fail test vẫn ổn, chỉ là warning.”

🟡 Mid 5) Incremental model hoạt động thế nào? unique_key dùng để làm gì? ▶

Gợi ý trả lời: incremental chỉ xử lý phần data mới/thay đổi thay vì rebuild full table mỗi lần. unique_key định danh record để engine MERGE/UPSERT đúng hàng, giúp idempotency khi retry/backfill.

Red flag: đừng nói... “Incremental nghĩa là insert thêm, không cần quan tâm duplicate/update.”

🟡 Mid 6) Tại sao cần surrogate key thay vì natural key từ source? ▶

Gợi ý trả lời: natural key từ source có thể null, đổi format, hoặc không globally unique giữa nhiều nguồn. Surrogate key chuẩn hóa join semantics, giảm coupling với upstream schema và ổn định cho marts/tests.

Red flag: đừng nói... “Source có ID rồi thì luôn đủ dùng, không cần surrogate key.”

🟡 Mid 7) dbt build khác dbt run thế nào? Khi nào dùng build? ▶

Gợi ý trả lời: dbt run chỉ materialize models; dbt build chạy seeds/snapshots/models/tests theo DAG, nên phù hợp CI/CD vì có quality gate tích hợp. Build thường là default command cho production deploy.

Red flag: đừng nói... “Run và build như nhau, chỉ khác tên lệnh.”

🟡 Mid 8) Làm sao đảm bảo dbt project chạy được ở cả DuckDB và BigQuery? ▶

Gợi ý trả lời: tách cấu hình theo target trong profiles.yml, gom logic đặc thù engine vào macros dispatch, hạn chế SQL vendor-specific trong models lõi. CI nên chạy ít nhất một subset test trên mỗi target quan trọng.

Red flag: đừng nói... “Copy-paste hai codebase SQL riêng là nhanh nhất.”

🔴 Senior 9) Pipeline incremental fail lúc 3h sáng sau khi merge 80% data — xử lý thế nào mà không mất data? ▶

Gợi ý trả lời: đóng băng downstream publish, xác định phạm vi commit dang dở bằng watermark/partition, rerun idempotent theo window affected. Nếu schema drift lớn, dùng snapshot backup + controlled full-refresh/backfill theo partition thay vì truncate toàn bộ.

Red flag: đừng nói... “Drop table rồi chạy lại full luôn cho chắc.”

🔴 Senior 10) Team 10 AE cùng dbt project, CI/CD setup thế nào để tránh conflict và tốn kém compute? ▶

Gợi ý trả lời: áp dụng Slim CI (state:modified+ + defer to prod), bắt buộc PR checks (compile/build/test/docs), ownership theo folder/domain và policy merge window. Dùng ephemeral dev schemas theo user để tránh dẫm nhau.

Red flag: đừng nói... “Mỗi PR chạy full 200 models trên prod warehouse cho an toàn.”

🔴 Senior 11) Thiết kế data model cho taxi trips support cả real-time dashboard lẫn monthly reporting — trade-offs? ▶

Gợi ý trả lời: tách serving layer theo latency requirement: near-real-time aggregate table cho dashboard và curated marts ổn định cho reporting tháng. Trade-off chính là freshness vs consistency/cost; cần định nghĩa SLA khác nhau và reconciliation job để đồng bộ số liệu.

Red flag: đừng nói... “Một bảng duy nhất giải quyết hết mọi use case BI.”

🔴 Senior 12) dbt project có 200 models, test coverage 30% — ưu tiên viết test theo thứ tự nào, tại sao? ▶

Gợi ý trả lời: ưu tiên theo risk x blast radius: (1) marts/business-critical KPIs, (2) keys/relationships của fact-dim, (3) freshness/contract cho sources, rồi mới mở rộng xuống staging chi tiết. Mục tiêu là giảm xác suất incident nặng trước khi tăng coverage tuyệt đối.

Red flag: đừng nói... “Viết random test cho đủ phần trăm coverage trước đã.”
