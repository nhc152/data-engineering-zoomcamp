TAB GROUP 1: download_data.sh
─────────────────────────────

Tab 1 — Code (nguyên văn):
```bash
set -e

TAXI_TYPE=$1 # "yellow"
YEAR=$2 # 2020

URL_PREFIX="https://github.com/DataTalksClub/nyc-tlc-data/releases/download"

for MONTH in {1..12}; do
  FMONTH=`printf "%02d" ${MONTH}`
  URL="${URL_PREFIX}/${TAXI_TYPE}/${TAXI_TYPE}_tripdata_${YEAR}-${FMONTH}.csv.gz"
  LOCAL_PREFIX="data/raw/${TAXI_TYPE}/${YEAR}/${FMONTH}"
  LOCAL_FILE="${TAXI_TYPE}_tripdata_${YEAR}_${FMONTH}.csv.gz"
  LOCAL_PATH="${LOCAL_PREFIX}/${LOCAL_FILE}"
  echo "downloading ${URL} to ${LOCAL_PATH}"
  mkdir -p ${LOCAL_PREFIX}
  wget ${URL} -O ${LOCAL_PATH}
done
```

Tab 2 — Phân tích:
- Arguments: $1=TAXI_TYPE, $2=YEAR
- set -e: dừng ngay khi lệnh fail
- printf "%02d": format month 2 chữ số (01..12)
- Output path: data/raw/{type}/{year}/{month}/{type}_tripdata_{year}_{month}.csv.gz
- Cách chạy: bash download_data.sh yellow 2020

Tab 3 — Điểm yếu (warn box):
- Không checksum: file download một nửa không phát hiện được
- Không skip-if-exists: chạy lại sẽ download lại toàn bộ
- Không retry/backoff: network lỗi → job fail ngay
- Hardcode URL domain

Bước 4: So sánh infer schema vs define schema
```python
from pyspark.sql import types

schema = types.StructType([
    types.StructField('hvfhs_license_num', types.StringType(), True),
    types.StructField('dispatching_base_num', types.StringType(), True),
    types.StructField('pickup_datetime', types.TimestampType(), True),
    types.StructField('dropoff_datetime', types.TimestampType(), True),
    types.StructField('PULocationID', types.IntegerType(), True),
    types.StructField('DOLocationID', types.IntegerType(), True),
    types.StructField('SR_Flag', types.StringType(), True)
])

df = spark.read \
    .option("header", "true") \
    .schema(schema) \
    .csv("fhvhv_tripdata_2021-01.csv")

df.printSchema()
```

[Expected output box]: schema hiện đúng types, không phải toàn StringType

[toggleSolution] — Điểm hay cần chú ý:
pickup_datetime/dropoff_datetime → TimestampType (không phải StringType)
PULocationID/DOLocationID → IntegerType
SR_Flag vẫn là String vì chỉ có giá trị "Y" hoặc null

[Catalyst Optimizer — 4 phases]

[Executor Memory Model]
Executor JVM heap:
- Execution memory: shuffle, join, sort (spark.executor.memory * 0.6 * 0.5)
- Storage memory: cache/persist (spark.executor.memory * 0.6 * 0.5)
- User memory: UDF, internal data structures
- Overhead: off-heap (spark.executor.memoryOverhead, default max(384MB, 10%))

Q1: Spark khác Hadoop MapReduce ở điểm gì?
A: Spark xử lý trong memory, chỉ ghi disk khi cần (shuffle/spill); MapReduce ghi disk sau mỗi step. Spark nhanh hơn 10-100x với iterative algorithms. Spark có DataFrame/SQL/Streaming API; MR chỉ có map và reduce primitives.

Q2: Transformation và Action trong Spark khác gì nhau? Cho ví dụ.
A: Transformation (lazy): map, filter, select, groupBy, join — chỉ build DAG, chưa execute. Action (trigger execution): count(), show(), collect(), write(). Spark không chạy gì cho đến khi gặp action.

Q3: Parquet tốt hơn CSV ở điểm nào cho batch processing?
A: Columnar storage — chỉ đọc columns cần; row group statistics cho predicate pushdown; built-in compression (Snappy/Gzip); self-describing schema; splittable để đọc parallel.

Q4: repartition và coalesce khác gì nhau? Khi nào dùng cái nào?
A: repartition(n) — full shuffle, tạo n partitions đồng đều — dùng khi muốn tăng partitions hoặc phân phối lại đều. coalesce(n) — narrow operation, không shuffle, merge partitions — dùng khi giảm partitions (như trước write), nhanh hơn nhưng có thể uneven.

─────────────────────────────
[Mid — badge vàng]
─────────────────────────────

Q5: Khi nào nên dùng DataFrame API, khi nào Spark SQL?
A: DataFrame API tốt hơn cho: chaining transformations phức tạp, type-checking lúc compile, programmatic column selection (ví dụ: dynamic column list). Spark SQL tốt hơn cho: aggregation phức tạp với nhiều GROUP BY, team quen SQL hơn Python, reuse query logic. Về performance, cả hai compile xuống cùng physical plan — không có winner tuyệt đối.

Q6: Giải thích data flow từ CSV raw đến BigQuery trong module này.
A: download_data.sh → raw CSV.gz trên local → 05_taxi_schema.ipynb (define schema, repartition 4, write Parquet) → data/pq/ → 06_spark_sql.py (union green/yellow, aggregate revenue, coalesce 1, write Parquet) → gsutil upload lên GCS → 06_spark_sql_big_query.py trên Dataproc (read GCS Parquet, write BigQuery qua connector) → BigQuery table trips_data_all.reports-YYYY

Q7: Tại sao registerTempTable bị deprecated và nên dùng gì thay?
A: registerTempTable bị deprecated từ Spark 2.0. Dùng createOrReplaceTempView('name') hoặc createGlobalTempView('name'). Global temp view tồn tại qua SparkSession, local temp view chỉ trong session.

[Senior — badge đỏ]
─────────────────────────────

Q8: Giải thích tại sao UDF trong PySpark chậm. Có giải pháp nào?
A: Python UDF phá vỡ Catalyst optimization (không pushdown), phải serialize mỗi row: JVM → Python via Py4J → process → Python → JVM. Giải pháp theo thứ tự ưu tiên: (1) dùng built-in pyspark.sql.functions thay UDF; (2) Pandas UDF (vectorized, Arrow-based) — nhanh hơn Python UDF ~10x; (3) Scala UDF nếu có thể; (4) Nếu bắt buộc Python UDF, batch nhiều operations trong 1 UDF.

Q9: Bạn sẽ diagnose và fix data skew như thế nào?
A: Detect: Spark UI → Stages → task duration — một số tasks chạy 10x lâu hơn median. Root cause: một số keys xuất hiện quá nhiều (ví dụ NULL, "unknown", hot users). Fix options: (1) Filter hoặc handle NULL riêng trước join/groupBy; (2) AQE (spark.sql.adaptive.skewJoin.enabled=true); (3) Salting: thêm random salt vào key để phân tán, replicate dimension table với salt; (4) Repartition by column khác. Trade-off: salting phức tạp hơn, cần join lại sau.

Q10: Design một production batch pipeline xử lý taxi data monthly. Cần gì ngoài những gì repo dạy?
A: Orchestration (Airflow/Prefect) với DAG per month; Idempotent partitioned writes (overwrite by month); Config/secrets management (Secret Manager, không hardcode); Schema evolution strategy (backward compatible); Data quality gates (row count, null rates, value ranges) trước khi promote to prod; Structured logging với run_id, input_partition, row_counts; Alerting khi job fail hoặc row count anomaly; Cost controls (spot workers, autoscaling, cluster auto-delete); CI/CD cho PySpark code; Data lineage tracking; SLA monitoring.
