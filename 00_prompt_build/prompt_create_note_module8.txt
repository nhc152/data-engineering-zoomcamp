===============================
1. CẤU TRÚC THƯ MỤC
===============================
Tree đầy đủ + mục đích từng folder và file.

Phải làm rõ cấu trúc chi tiết của cohorts:
- 2022: cohort cũ theo week, có Airflow examples cho ingestion và warehouse.
- 2023: cohort theo week, có homework, leaderboard, project, streaming Confluent/Spark example.
- 2024: cohort theo module mới, có homework/solutions, workshop dlt, RisingWave, notebook.
- 2025: cohort theo module mới, có Kestra YAML flows, DLT upload, workshop dlt, streaming notebook.
- 2026: cohort mới nhất, có module 05 data platforms, dlt workshop cập nhật, notebook và scripts.

Phải làm rõ cấu trúc chi tiết của projects:
- projects/README.md: final project requirements, rubric, peer review, submission rules, tips, portfolio guidance.
- projects/datasets.md: gợi ý dataset cho final project.

Phải làm rõ cấu trúc chi tiết của images:
- images/architecture: architecture/course overview/workshop architecture images.
- images/aws: AWS IAM image used by older cohort docs.
- root images: logos/assets for tools and workshops.

Với mỗi năm, ghi:
- Các module/week có trong năm đó.
- File nào là README/project/leaderboard.
- File nào là homework/solution.
- File nào là code/config/notebook/image.
- Những nội dung bị lặp giữa các năm và những nội dung thay đổi theo năm.

Với projects/images, ghi:
- File nào được README hoặc cohort docs tham chiếu.
- Asset nào phục vụ course overview, workshop, hay project guidance.
- File ảnh nào chỉ là static asset, không có logic/code.

===============================
2. NỘI DUNG CÁC FILE .MD
===============================
Tóm tắt từng file .md theo format:
[Tên file đầy đủ theo path]
- Vai trò: README / homework / solution / project / workshop / setup guide / concepts.
- Dạy về: [topic]
- Concepts chính: [list]
- Commands được dùng: [list commands thực tế, nếu có]
- Dataset/table/service liên quan: [nếu có]
- Câu hỏi homework chính: [nếu là homework]
- Đáp án/solution chính: [nếu là solution]
- Lưu ý đặc biệt: [nếu có]

Không gộp các file cùng tên ở các năm khác nhau. Ví dụ:
- cohorts/2024/01-docker-terraform/homework.md
- cohorts/2025/01-docker-terraform/homework.md
- cohorts/2026/01-docker-terraform/homework.md
phải được phân tích riêng.

===============================
3. PHÂN TÍCH CODE FILES & CONFIGS
===============================
Với mỗi file code/config, ghi rõ:

[Python files .py]
- Mục đích file.
- Inputs/arguments/env vars.
- Libraries dùng.
- Data source và sink.
- Logic chính từng bước.
- File nào gọi/chạy file này.
- Copy nguyên văn toàn bộ file.

[YAML/YML files: docker-compose, Kestra flows, configs]
- Mục đích file.
- Services/flows/tasks.
- Images/build, ports, volumes, env vars, networks, depends_on nếu là Docker Compose.
- Namespace, tasks, triggers, inputs, outputs nếu là Kestra flow.
- External dependencies: Postgres, GCS, BigQuery, dbt, Kafka/Redpanda, Spark...
- Copy nguyên văn toàn bộ file.

[Shell files .sh]
- Mục đích file.
- Commands chính.
- Arguments/env vars.
- Cách chạy.
- Copy nguyên văn toàn bộ file.

[Dockerfile / requirements.txt / pyproject.toml / client.properties / .env_example]
- Dependencies/config quan trọng.
- Version/package quan trọng.
- Credentials/secrets/config nhạy cảm nếu có.
- Copy nguyên văn toàn bộ file.

[Notebook .ipynb]
- Mục đích notebook.
- Packages/imports.
- Các cell chính và logic.
- Dataset/source/sink.
- Không cần copy toàn bộ JSON notebook nếu quá dài; nhưng phải tóm tắt từng cell quan trọng và copy các đoạn code cell chính.

[Image/binary files .png/.jpg]
- Không copy nội dung nhị phân.
- Ghi rõ path, vai trò, hình minh họa gì, file .md/.ipynb nào tham chiếu nếu xác định được.

===============================
8. ĐIỂM YẾU & THIẾU SÓT
===============================
Phân tích dựa trên code/config/thư mục thực tế:
- Security issues: credentials, client.properties, env vars, hardcoded connection string.
- Idempotency issues: rerun homework/pipeline có bị duplicate hoặc fail không.
- Production readiness gaps: deployment, retries, rollback, scaling, partitioning.
- Data quality gaps: validation, schema checks, null handling, tests.
- Observability gaps: logs, metrics, alerts, lineage.
- Testing gaps: unit/integration tests, workflow tests, DAG tests, notebook reproducibility.
- Cost/performance gaps: BigQuery partitioning/clustering, Spark tuning, Kafka retention/partitioning.
- Documentation gaps: phần nào chỉ là homework prompt, chưa đủ hướng dẫn production.
- Project guidance gaps: rubric có yêu cầu gì nhưng chưa có template/reference implementation.
- Asset gaps: ảnh nào thiếu context, ảnh nào được dùng nhưng không có mô tả đầy đủ.
- Những gì cohorts dạy nhưng chưa đủ cho Senior Data Engineer.

===============================
9. KIẾN THỨC NGOÀI REPO
===============================
Dựa trên những gì cohorts đang cover, liệt kê những concept Senior Data Engineer phải biết nhưng repo KHÔNG dạy đủ:
- Docker/Terraform production: remote state, modules, secrets, image scanning, CI/CD.
- Orchestration: Airflow vs Kestra internals, backfills, retries, SLAs, event-driven orchestration.
- Warehouse: BigQuery internals, partitioning/clustering strategy, cost governance, data modeling.
- Analytics engineering: dbt testing, snapshots, exposures, contracts, semantic layer, CI.
- Batch: Spark internals, shuffle, partitioning, file layout, lakehouse formats.
- Streaming: Kafka internals, consumer groups, schema registry, exactly-once, Flink/Spark state.
- Data quality: Great Expectations/Soda/dbt tests, anomaly detection, contracts.
- Governance: IAM, PII, catalog, lineage, access control.
- Platform engineering: multi-env, IaC, monitoring, incident response.
- Project practice: requirements, architecture docs, SLAs, runbooks, evaluation criteria.
- Portfolio/project delivery: README structure, reproducibility, demo data, CI, deployment docs, architecture diagrams.
