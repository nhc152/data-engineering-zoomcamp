WHYVấn đề Batch Processing là gì?▼

Tại sao không dùng pandas thuần?

Pandas chạy rất tốt khi dữ liệu vừa với RAM của một máy và logic xử lý không cần phân tán. Khi dữ liệu tăng lên hàng chục GB, hàng trăm GB hoặc cần backfill nhiều tháng lịch sử, single-machine limit trở thành bottleneck: RAM đầy, CPU một máy không đủ, và script tuần tự chạy quá lâu.

Trước Spark người ta làm gì?

  • Hadoop MapReduce: bền và scale tốt nhưng phải ghi đọc disk nhiều vòng, code dài và chậm cho workflow lặp.
  • Hive: tiện viết SQL trên Hadoop nhưng latency thường cao, phù hợp batch lớn hơn là exploration nhanh.
  • Sequential scripts: Python/bash chạy lần lượt, dễ bắt đầu nhưng khó scale, khó retry từng partition, dễ đụng giới hạn RAM.

Spark giải quyết gì?

Spark là engine xử lý phân tán in-memory. Bạn mô tả pipeline bằng transformations, Spark giữ kế hoạch dưới dạng DAG, tối ưu lazy evaluation bằng Catalyst, rồi chia thành stages và tasks chạy trên executors khi action được gọi.

Mental model: Spark giúp biến một recipe xử lý dữ liệu lớn thành nhiều task nhỏ chạy song song trên cluster.

Spark Architecture

Driver program là process chạy main application, giữ SparkSession/SparkContext, build DAG và điều phối job. SparkContext là cửa ngõ nói chuyện với cluster manager. Cluster Manager cấp tài nguyên: local mode, Standalone, YARN, Kubernetes hoặc Dataproc-managed cluster. Executors là process chạy trên worker nodes, thực thi tasks và giữ cache/shuffle data. Task là đơn vị chạy nhỏ nhất trên một partition.

  • Local mode: mọi thứ chạy trên máy local, ví dụ local[*] dùng tất cả CPU cores khả dụng.
  • Standalone: Spark tự quản master/worker.
  • YARN: Hadoop cluster manager, phổ biến trong Hadoop ecosystem.
  • Kubernetes: chạy driver/executor dưới dạng pods.

Lazy Evaluation

Transformations như map, filter, select, groupBy, join chỉ xây execution plan. Actions như count, show, write, collect mới trigger execution. Lazy evaluation cho phép Catalyst nhìn toàn bộ DAG, tối ưu projection/filter pushdown, join strategy và physical plan trước khi chạy.

Spark = distributed lazy computation engine — bạn define recipe, Spark optimize và cook khi cần.

Narrow vs Wide Dependencies

Narrow không cần shuffle: map, filter, union. Dữ liệu của output partition đến từ một hoặc vài input partition cố định. Wide cần shuffle: groupBy, join, repartition. Dữ liệu phải di chuyển giữa partitions.

P2 -> map/filter -> P2'
P3 -> map/filter -> P3'

Wide:
P1 --\

Partitions & Shuffle

  • repartition(n): full shuffle, phân phối đều hơn, đắt hơn.
  • coalesce(n): giảm số partition với ít shuffle hơn, nhanh hơn nhưng dễ uneven.
  • spark.sql.shuffle.partitions: mặc định 200, ảnh hưởng số partition sau shuffle SQL/DataFrame.
Production concern: repo dùng repartition(4) và coalesce(1) hardcoded. Với data/cluster khác, 4 partitions có thể quá ít hoặc quá nhiều; coalesce(1) ép single file, tạo bottleneck, phá parallelism và dễ OOM ở task cuối.

Senior box — Catalyst & Tungsten

Catalyst pipeline: Unresolved Logical Plan -> Analyzed Logical Plan -> Optimized Logical Plan -> Physical Plan -> RDD execution. Tungsten tập trung whole-stage codegen, off-heap memory và binary processing để giảm overhead object JVM. Khi debug performance, dùng df.explain(True) để xem parsed/analyzed/optimized/physical plan, filter pushdown, join strategy và shuffle boundary.

Tại sao Parquet?

  • Columnar storage: query SELECT col1, col2 chỉ đọc columns cần thiết.
  • Row group statistics: min/max/null count hỗ trợ predicate pushdown.
  • Built-in compression: Snappy/Gzip/Zstd giúp giảm storage và scan bytes.
  • Native support: Spark, BigQuery, Athena đều đọc tốt Parquet.

download_data.shnguyên văn
set -e

TAXI_TYPE=$1 # "yellow"
YEAR=$2 # 2020

URL_PREFIX="https://github.com/DataTalksClub/nyc-tlc-data/releases/download"

for MONTH in {1..12}; do
  FMONTH=`printf "%02d" ${MONTH}`
  URL="${URL_PREFIX}/${TAXI_TYPE}/${TAXI_TYPE}_tripdata_${YEAR}-${FMONTH}.csv.gz"
  LOCAL_PREFIX="data/raw/${TAXI_TYPE}/${YEAR}/${FMONTH}"
  LOCAL_FILE="${TAXI_TYPE}_tripdata_${YEAR}_${FMONTH}.csv.gz"
  LOCAL_PATH="${LOCAL_PREFIX}/${LOCAL_FILE}"
  echo "downloading ${URL} to ${LOCAL_PATH}"
  mkdir -p ${LOCAL_PREFIX}
  wget ${URL} -O ${LOCAL_PATH}
done
  • Arguments: $1=TAXI_TYPE, $2=YEAR.
  • set -e: dừng ngay khi một lệnh fail.
  • printf "%02d": format month 2 chữ số 01..12.
  • Output path: data/raw/{type}/{year}/{month}/{type}_tripdata_{year}_{month}.csv.gz.
  • Cách chạy: bash download_data.sh yellow 2020.
Điểm yếu: Không checksum nên file download một nửa có thể không bị phát hiện; không skip-if-exists nên chạy lại tải lại toàn bộ; không retry/backoff nên network lỗi là job fail ngay; URL domain bị hardcode.

Interview Prep

JRJunior▼
Q1: Spark khác Hadoop MapReduce ở điểm gì?▶

Spark xử lý in-memory, chỉ ghi disk khi cần như shuffle/spill; MapReduce thường ghi disk sau mỗi step. Spark nhanh hơn nhiều với iterative algorithms và có DataFrame/SQL/Streaming API; MapReduce chỉ có map/reduce primitives.

Q2: Transformation và Action trong Spark khác gì nhau? Cho ví dụ.▶

Transformation là lazy: map, filter, select, groupBy, join chỉ build DAG. Action trigger execution: count(), show(), collect(), write(). Spark không chạy gì cho đến khi gặp action.

Q3: Parquet tốt hơn CSV ở điểm nào cho batch processing?▶

Parquet là columnar nên chỉ đọc columns cần; có row group statistics cho predicate pushdown; có built-in compression như Snappy/Gzip; self-describing schema; splittable để đọc parallel.

Q4: repartition và coalesce khác gì nhau? Khi nào dùng cái nào?▶

repartition(n) full shuffle, tạo n partitions đồng đều, dùng khi muốn tăng partitions hoặc phân phối lại đều. coalesce(n) thường narrow operation, merge partitions để giảm số partitions, nhanh hơn nhưng có thể uneven.

Q5: Khi nào nên dùng DataFrame API, khi nào Spark SQL?▶

DataFrame API tốt cho chaining transformations phức tạp, programmatic column selection và logic gắn với code. Spark SQL tốt cho aggregation phức tạp, team quen SQL và reuse query logic. Về performance, cả hai compile xuống physical plan, nên không có winner tuyệt đối.

Q6: Giải thích data flow từ CSV raw đến BigQuery trong module này.▶

download_data.sh tải raw CSV.gz local -> 05_taxi_schema.ipynb define schema, repartition 4, write Parquet -> data/pq/ -> 06_spark_sql.py union green/yellow, aggregate revenue, coalesce 1, write Parquet -> upload GCS -> 06_spark_sql_big_query.py trên Dataproc đọc GCS Parquet, ghi BigQuery qua connector -> trips_data_all.reports-YYYY.

Q7: Tại sao registerTempTable bị deprecated và nên dùng gì thay?▶

registerTempTable bị deprecated từ Spark 2.0. Dùng createOrReplaceTempView('name') hoặc createGlobalTempView('name'). Global temp view tồn tại qua SparkSession, local temp view chỉ trong session.

Q8: Giải thích tại sao UDF trong PySpark chậm. Có giải pháp nào?▶

Python UDF phá vỡ Catalyst optimization và phải serialize mỗi row qua JVM -> Python via Py4J -> Python process -> JVM. Giải pháp theo thứ tự: dùng built-in pyspark.sql.functions; dùng Pandas UDF vectorized Arrow-based; dùng Scala UDF nếu có thể; nếu bắt buộc Python UDF thì batch nhiều operations trong một UDF.

Q9: Bạn sẽ diagnose và fix data skew như thế nào?▶

Detect trong Spark UI -> Stages -> task duration, khi vài tasks chạy lâu hơn median rất nhiều. Root cause thường là hot keys như NULL, unknown, hot users. Fix: xử lý NULL riêng, bật AQE spark.sql.adaptive.skewJoin.enabled=true, salting key để phân tán và replicate dimension table, hoặc repartition theo key khác. Salting hiệu quả nhưng phức tạp hơn.

Q10: Design một production batch pipeline xử lý taxi data monthly. Cần gì ngoài repo?▶

Cần orchestration với DAG per month, idempotent partitioned writes, config/secrets management, schema evolution, data quality gates, structured logging với run_id/input_partition/row_counts, alerting khi fail hoặc row count anomaly, cost controls như spot workers/autoscaling/auto-delete, CI/CD cho PySpark code, lineage tracking và SLA monitoring.
