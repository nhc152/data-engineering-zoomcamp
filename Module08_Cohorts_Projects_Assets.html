WHY Module 08 giải quyết vấn đề gì? ▼

Đây không phải module kỹ thuật lõi

cohorts/, projects/, và images/ không giống 01-docker-terraform hay 07-streaming. Chúng là lớp course wrapper: lịch sử khóa học, bài tập theo năm, hướng dẫn final project, và tài sản hình ảnh.

Mental model: Module 01-07 dạy công cụ và pipeline. Module 08 dạy cách dùng repo như một bản đồ học tập và bản thiết kế portfolio.

3 folder chính

FolderVai trò trong repoGiá trị thực tế
cohorts/Lịch sử course 2022-2026, homework, workshops, project links.Chọn đúng homework theo năm, đọc lại tool cũ như Airflow 2022 hoặc Kestra 2025.
projects/Final project requirements, rubric, datasets, peer review, portfolio advice.Tự chấm capstone trước khi submit hoặc đưa vào CV.
images/Static assets: architecture diagrams, logos, AWS screenshot.Biết ảnh nào phục vụ README, workshop, hoặc project diagram.

Vấn đề thực tế

  • Người học dễ chỉ đọc cohort mới nhất và bỏ qua homework/workshop hữu ích ở cohort cũ như Airflow 2022 hoặc Confluent/Spark 2023.
  • Final project thường fail không phải vì thiếu tool, mà vì thiếu rubric, dataset strategy, reproducibility và documentation.
  • Assets/images bị dùng rải rác trong README/cohorts; nếu không inventory, HTML hoặc project README dễ có broken link hoặc ảnh sai context.
Cảnh báo: Không bao giờ ghi "homework.md" trống context. Luôn dùng full path như cohorts/2026/03-data-warehouse/homework.md.

OK Exit criteria ▼
  • Giải thích được vai trò từng folder cohorts/, projects/, images/.
  • Biết nên học cohort nào cho từng chủ đề: latest homework, Airflow cũ, Kestra flows, dlt workshop, project examples.
  • Đọc được project rubric và tự đánh giá capstone project theo problem, cloud/IaC, ingestion, DWH, transformation, dashboard, reproducibility.
  • Biết NYC taxi dataset không được dùng cho final project và biết cách chọn dataset thay thế.
  • Biết ảnh/logo nào được dùng ở README/cohort/workshop nào, và không copy binary vào notes.
  • Biết chuyển course materials thành portfolio project có checklist, README, architecture diagram và reproducible commands.

Dataset rule: NYC taxi dataset dùng xuyên suốt course và homework, nên không được dùng cho final project. Chọn dataset khác.
Mental model: projects/README.md là contract giao hàng. Bạn không bắt đầu final project từ tool; bạn bắt đầu từ rubric, problem statement và dataset.

EX Project design examples & anti-patterns ▼

Design examples

LevelDesignKhi nào hợp lý
BeginnerBatch ELT public API -> GCS/BigQuery -> dbt -> dashboard.Khi dataset update daily/hourly và bạn cần project stable.
IntermediateScheduled ingestion + partitioned warehouse + dbt tests.Khi muốn chấm cao ở orchestration, DWH optimization và transformations.
AdvancedStreaming/CDC hoặc dlt + orchestration + monitoring.Khi data có real-time value hoặc API ingestion phức tạp.

Anti-patterns

  • Hardcoded local paths hoặc personal cloud project IDs.
  • Không chạy thử docker compose up trước khi submit.
  • Thiếu dbt deps khi dùng dbt packages.
  • README chỉ có screenshot, không có commands.
  • Không có architecture diagram.
  • Reuse dataset bị cấm như NYC taxi course dataset.

D Interview/project defense questions ▼
1. Vì sao bạn chọn dataset này?

Answer guide: Gắn dataset với problem statement, public access, size, schema, update cadence và risk thấp.

Evidence: Link dataset, schema sample, README dataset section.

Red flag: Chọn dataset vì dễ copy hoặc dùng lại NYC taxi course dataset.

2. Dataset có update cadence không?

Answer guide: Nêu daily/hourly/monthly/static và cadence đó quyết định batch/stream schedule thế nào.

Evidence: Dataset docs, API metadata, timestamp columns, ingestion schedule.

Red flag: Schedule tùy ý không liên quan source.

3. Pipeline có idempotent không?

Answer guide: Mô tả partition/date key, staging table, merge/upsert hoặc overwrite an toàn.

Evidence: Flow/DAG code, SQL merge, object path convention.

Red flag: Rerun tạo duplicate hoặc không biết rerun sẽ xảy ra gì.

4. Nếu ingestion fail giữa chừng thì recover thế nào?

Answer guide: Có retry, checkpoint, staging cleanup, rerun theo partition và log lỗi.

Evidence: Orchestration config, runbook, error handling code.

Red flag: Phải xóa thủ công toàn bộ warehouse để chạy lại.

5. Vì sao chọn warehouse/lakehouse này?

Answer guide: Liên hệ query pattern, data size, cloud stack, cost và skill target.

Evidence: Architecture diagram, table design, cost/region note.

Red flag: Chọn vì course dùng mà không giải thích project fit.

6. Partitioning/clustering/file layout thế nào?

Answer guide: Nêu partition key, clustering columns hoặc object layout theo date/source.

Evidence: DDL, dbt model config, storage paths.

Red flag: Full scan mọi query hoặc file nhỏ không kiểm soát.

7. Transformations được test ra sao?

Answer guide: Có dbt tests, SQL assertions, schema checks, null/range checks hoặc validation script.

Evidence: schema.yml, test command, validation output.

Red flag: Chỉ nhìn dashboard rồi kết luận data đúng.

8. Dashboard trả lời business question gì?

Answer guide: Mỗi tile phải gắn với câu hỏi và quyết định/action cụ thể.

Evidence: Dashboard screenshot/link, metric definitions, README explanation.

Red flag: Chart đẹp nhưng không trả lời câu hỏi nào.

9. Project có reproducible từ máy khác không?

Answer guide: Clean clone, env setup, docker compose, dbt deps, seed/sample data và commands theo thứ tự.

Evidence: README setup section, .env.example, compose/IaC files.

Red flag: Chỉ chạy được trên laptop tác giả.

10. Secrets được quản lý thế nào?

Answer guide: Dùng env vars, secret manager hoặc local ignored files; README chỉ hướng dẫn placeholders.

Evidence: .gitignore, .env.example, secret docs.

Red flag: Commit credential, token, service account hoặc hardcoded personal project ID.

11. Peer reviewer cần chạy command nào đầu tiên?

Answer guide: Nêu command bootstrap đầu tiên và prerequisite rõ.

Evidence: README quickstart, Makefile/script, docker compose command.

Red flag: Reviewer phải đoán thứ tự chạy.

12. Bạn dùng rubric để tự chấm ra sao?

Answer guide: Tự chấm từng area: problem, cloud, ingestion, warehouse, transformations, dashboard, reproducibility.

Evidence: Rubric self-check table trong README hoặc project notes.

Red flag: Không biết rubric có những hạng mục nào.

13. Phần nào lấy cảm hứng từ cohort cũ?

Answer guide: Nêu rõ full path và chỉ ra bạn học pattern gì, không copy mù.

Evidence: Reference section: Airflow 2022, streaming 2023, dlt 2024/2026 hoặc Kestra 2025.

Red flag: Copy solution/code mà không hiểu dataset/table.

14. Nếu đổi tool orchestration thì phần nào thay đổi?

Answer guide: Tách business logic khỏi scheduler; đổi DAG/flow wrapper, retry, secrets, schedule và deployment.

Evidence: Pipeline modules, orchestration files, config boundaries.

Red flag: Logic ingestion bị khóa chặt trong UI/tool cụ thể.

15. Điều gì làm project này đủ portfolio?

Answer guide: Project reviewable, reproducible, defensible: problem rõ, dataset hợp lệ, architecture thật, commands chạy được, insights có ý nghĩa.

Evidence: README, demo/dashboard, tests, diagram, self-review rubric.

Red flag: Chỉ là homework copy hoặc notebook rời rạc.
