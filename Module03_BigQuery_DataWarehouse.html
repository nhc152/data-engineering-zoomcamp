Vấn đề gốc: analytics query làm nghẽn OLTP

Trong hệ thống transactional, app cần phản hồi nhanh cho user (insert/update đơn lẻ, latency thấp). Khi team BI chạy query scan nhiều tháng dữ liệu ngay trên OLTP DB, CPU/IO bị tranh chấp, lock tăng, dashboard có thể xong nhưng production app thì chậm hoặc timeout.

🔥 Nguy hiểm: "Tiện tay query luôn DB production" là anti-pattern kinh điển; hệ quả là user-facing workload trả giá cho nhu cầu analytics.

Tại sao phải tách DW khỏi OLTP?

  • Workload isolation: OLTP tối ưu cho write nhỏ, OLAP tối ưu cho scan lớn.
  • Schema strategy khác nhau: OLTP thường normalized để tránh anomaly; DW thường denormalized/star để đọc nhanh.
  • Cost control: với BigQuery, bạn theo dõi bytes scanned theo query thay vì nuôi cluster cố định.
💡 Tip: Mục tiêu của DW không chỉ "lưu nhiều", mà là tách đúng loại bài toán để từng hệ thống làm đúng việc của nó.

Trước BigQuery, người ta làm gì?

Nhiều team dùng on-prem Hadoop/Hive hoặc tự quản cluster DWH (kể cả Redshift đời đầu). Bài toán lớn không phải chỉ SQL, mà là vận hành cluster: sizing, capacity planning, tuning storage/compute, backup, nâng cấp version, xử lý node fail. BigQuery giảm tải phần vận hành đó bằng mô hình serverless.

⚠️ Cảnh báo: Serverless không có nghĩa là "không cần design"; nếu query kém hoặc partition sai, chi phí vẫn đội rất nhanh.

Beginner — Data Warehouse vs Data Lake vs Data Lakehouse

  • Data Warehouse: Dữ liệu đã có schema, tối ưu SQL analytics, governance rõ. Dùng khi KPI/dashboard là trọng tâm.
  • Data Lake: Lưu raw data đa định dạng (CSV/JSON/Parquet/Avro). Dùng khi cần lưu trữ linh hoạt, khám phá dữ liệu, data science.
  • Lakehouse: Kết hợp linh hoạt lake + quản trị/ACID gần warehouse. Dùng khi muốn một nền tảng cho cả BI lẫn ML với volume rất lớn.
📌 Lưu ý: Chọn kiến trúc theo workload và team maturity, không theo trend tool.

Intermediate — Column-oriented storage: vì sao BigQuery không row-store?

Phân tích dữ liệu thường chỉ chạm vài cột nhưng trên rất nhiều dòng. Row-store buộc đọc cả row (nhiều cột không cần), còn columnar chỉ đọc cột cần thiết nên giảm I/O mạnh. Đó là lý do BigQuery tối ưu tốt cho SELECT colA, SUM(colB) ... trên hàng tỷ rows.

💡 Tip: Với BigQuery, số cột bạn chọn ảnh hưởng trực tiếp bytes scanned; tránh SELECT * trừ khi đang khám phá data ban đầu.

Senior — BigQuery storage internals & execution

  • Capacitor format: engine cột nén cao, tối ưu scan theo column chunk.
  • Execution tree / stages: query được chia nhiều stage, mỗi stage xử lý song song rồi shuffle/aggregate theo execution plan.
  • Slot management: slot là đơn vị compute; query cạnh tranh slot nên thiết kế workload và lịch chạy ảnh hưởng latency/cost rõ rệt.
🔥 Nguy hiểm: Tuning SQL mà bỏ qua execution stages và slot contention thường dẫn tới tối ưu "ảo" (một query nhanh hơn nhưng cả hệ thống chậm hơn).

🧩 Partitioning & Clustering (Beginner → Senior) ▼

Beginner — Partition là gì?

Partition = chia table thành các "ngăn" theo một key (thường là ngày). Analogy: tủ hồ sơ chia theo năm/tháng; khi cần hồ sơ tháng 06/2019, bạn mở đúng ngăn đó thay vì lục toàn bộ tủ.

Beginner — Clustering là gì?

Clustering = sắp xếp dữ liệu trong từng partition theo một hoặc vài cột. Analogy: trong ngăn hồ sơ tháng 06, bạn sắp theo alphabet/vendor để tìm nhanh hơn.

Intermediate — Partition types: DATE / RANGE / INGESTION_TIME

  • DATE/TIMESTAMP partition: dùng khi query luôn lọc theo event date/business date.
  • RANGE partition: dùng khi cột số có miền rõ (id range, score range).
  • INGESTION_TIME partition: dùng khi chưa có event time sạch hoặc cần tracking thời điểm dữ liệu vào hệ thống.
⚠️ Cảnh báo: Chọn sai partition key sẽ khiến query phải scan nhiều partition, mất gần hết lợi ích thiết kế.

Intermediate — Chọn clustering key theo cardinality thế nào?

Thực tế nên ưu tiên cột có cardinality vừa đến cao và thường xuất hiện trong filter/join (ví dụ VendorID hoặc location IDs tùy workload). Cardinality quá thấp thường ít giúp prune; cardinality cực cao nhưng ít được filter cũng không hiệu quả.

Senior — Partition pruning mechanics

Pruning chỉ hiệu quả khi predicate "sargable" với partition column. Với table partition theo DATE(tpep_pickup_datetime), nên filter trực tiếp khoảng ngày:

● ● ●partition_pruning.sql
-- good: filter trực tiếp trên date range
WHERE DATE(tpep_pickup_datetime) BETWEEN '2019-06-01' AND '2019-06-30'

Tránh wrapping hàm phức tạp làm optimizer khó prune. Kiểm tra bytes processed để xác nhận pruning thực sự xảy ra.

Senior — Combined partition + cluster: thứ tự tác động

  • Partition giảm phạm vi dữ liệu theo thời gian trước (macro pruning).
  • Cluster tiếp tục giảm scan trong partition khi filter theo cluster key (micro pruning).

Từ benchmark trong repo big_query.sql:

  • Non-partitioned query: scan khoảng 1.6GB.
  • Partitioned query: scan khoảng ~106MB cho cùng time window.
  • Partitioned (không cluster) vs partitioned+clustered cho query có VendorID: 1.1GB vs 864.5MB.
💡 Mental model: Partition cắt theo thời gian, Cluster sắp xếp trong partition theo dimension key.

● ● ●big_query.sql (excerpt)
CREATE OR REPLACE EXTERNAL TABLE `taxi-rides-ny.nytaxi.external_yellow_tripdata`
OPTIONS (
  format = 'CSV',
  uris = [
    'gs://nyc-tl-data/trip data/yellow_tripdata_2019-*.csv',
    'gs://nyc-tl-data/trip data/yellow_tripdata_2020-*.csv'
  ]
);

CREATE OR REPLACE TABLE taxi-rides-ny.nytaxi.yellow_tripdata_non_partitioned AS
SELECT * FROM taxi-rides-ny.nytaxi.external_yellow_tripdata;

CREATE OR REPLACE TABLE taxi-rides-ny.nytaxi.yellow_tripdata_partitioned
PARTITION BY DATE(tpep_pickup_datetime) AS
SELECT * FROM taxi-rides-ny.nytaxi.external_yellow_tripdata;

-- non-partitioned scan: ~1.6GB
SELECT DISTINCT(VendorID)
FROM taxi-rides-ny.nytaxi.yellow_tripdata_non_partitioned
WHERE DATE(tpep_pickup_datetime) BETWEEN '2019-06-01' AND '2019-06-30';

-- partitioned scan: ~106MB
SELECT DISTINCT(VendorID)
FROM taxi-rides-ny.nytaxi.yellow_tripdata_partitioned
WHERE DATE(tpep_pickup_datetime) BETWEEN '2019-06-01' AND '2019-06-30';
  • Wildcard GCS URI (`*_2019-*.csv`): gom nhiều files theo tháng vào một external table, không cần khai báo từng file.
  • Input → output: external table đọc trực tiếp object trên GCS; `CREATE TABLE AS SELECT` materialize thành native table trong BigQuery storage.
  • Khi nào scan data: external table scan file GCS lúc query; native table scan dữ liệu đã lưu nội bộ BQ khi query.
  • Vì sao materialize trước partition/cluster: tách rõ bước ingest và bước tối ưu schema/query; sau đó mới apply partition/cluster để benchmark kiểm soát.
  • Hiệu năng thực tế từ code: query cùng điều kiện ngày giảm từ khoảng 1.6GB xuống ~106MB nhờ partition pruning.

  DATA_SPLIT_METHOD='AUTO_SPLIT'
) AS
SELECT * FROM `taxi-rides-ny.nytaxi.yellow_tripdata_ml` WHERE tip_amount IS NOT NULL;

SELECT * FROM ML.EVALUATE(MODEL `taxi-rides-ny.nytaxi.tip_model`, (...));
SELECT * FROM ML.PREDICT(MODEL `taxi-rides-ny.nytaxi.tip_model`, (...));
SELECT * FROM ML.EXPLAIN_PREDICT(MODEL `taxi-rides-ny.nytaxi.tip_model`, (...), STRUCT(3 as top_k_features));
  • Flow chuẩn: feature table → create model → evaluate → predict → explain predict → tune hyperparameters.
  • Vì sao cast LocationID sang STRING: ID vị trí là categorical, không mang thứ tự tuyến tính; cast giúp BQML xử lý đúng kiểu category thay vì numeric scale.
  • `linear_reg` phù hợp khi: cần baseline regression dễ giải thích; không phù hợp quan hệ phi tuyến quá mạnh nếu không có feature engineering thêm.
  • `ML.EXPLAIN_PREDICT`: trả về độ đóng góp feature cho từng prediction, hữu ích để debug model behavior và giải thích cho stakeholder.
  • Hyperparameter tuning: num_trials=5 (số lần thử); max_parallel_trials=2 (chạy song song); l1_reg=hparam_range(0, 20) và l2_reg=hparam_candidates([0, 0.1, 1, 10]) dùng để regularization chống overfitting.
  • Tại sao cast PULocationID/DOLocationID sang STRING: Location ID là identifier, không có linear ordering (ID 1 không "nhỏ hơn" ID 2 theo nghĩa thực). Nếu giữ numeric, model dễ coi như continuous variable và học feature sai; cast STRING giúp BQML xử lý dạng categorical (one-hot encoding) đúng ngữ nghĩa.
  • ML.FEATURE_INFO trả về gì: thống kê phân phối feature như mean, stddev, null_count, category_count; dùng để soi skewed features và missing values trước khi train.

💡 Xem lời giải
● ● ●task3_solution.py
import hashlib, json, logging
from datetime import datetime

logger = logging.getLogger("web_to_gcs")
logger.setLevel(logging.INFO)
logger.addHandler(logging.StreamHandler())

def md5_file(path):
    h = hashlib.md5()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(8192), b""):
            h.update(chunk)
    return h.digest()

def log_event(event, **kwargs):
    payload = {"ts": datetime.utcnow().isoformat() + "Z", "event": event, **kwargs}
    logger.info(json.dumps(payload))

r = requests.get(request_url)
r.raise_for_status()  # (1) fail-fast HTTP safety
open(file_name, "wb").write(r.content)

upload_to_gcs(BUCKET, object_name, local_file)
local_md5 = md5_file(local_file)
blob = storage.Client().bucket(BUCKET).blob(object_name)
blob.reload()
remote_md5 = blob.md5_hash  # base64 md5 from GCS metadata

log_event(
    "upload_complete",
    file=local_file,
    size_bytes=os.path.getsize(local_file),
    local_md5_hex=local_md5.hex(),
    remote_md5_b64=remote_md5,
)

🧯 Phần 5 — Cố tình phá (6 kịch bản) + hướng dẫn debug ▼

1) External table trỏ vào bucket/path sai hoặc đã bị xóa

  • Lỗi thường thấy khi query: lỗi đọc object từ GCS (not found/permission denied).
  • Tìm full message trong BigQuery Job details → Execution details / Error tab.
  • Vì sao vẫn CREATE được: external table chỉ lưu metadata URI; dữ liệu được kiểm tra lúc query.

2) Partition by cột có NULL

  • Dữ liệu NULL đi vào partition đặc biệt `__NULL__`.
  • Kiểm tra qua `INFORMATION_SCHEMA.PARTITIONS` để thấy partition_id tương ứng.

3) Query partitioned table nhưng WHERE không dùng partition column

  • Hệ quả: scan lớn gần full table.
  • `YEAR(pickup_date)=2019` thường không pruning tốt bằng filter range trực tiếp trên date.
  • Nên dùng: DATE(pickup_date) BETWEEN '2019-01-01' AND '2019-12-31'.

4) Chạy big_query_ml.sql trước khi có yellow_tripdata_partitioned

  • Error thực tế: table not found cho nguồn feature.
  • Fix: quay lại chạy `big_query.sql` theo đúng thứ tự, verify bảng tồn tại rồi chạy lại ML script.

5) web_to_gcs.py fail giữa chừng (Ctrl+C sau 3/12 files)

  • Baseline script có thể upload lại/ghi đè tùy trạng thái file local/GCS.
  • Bản progress-bar an toàn hơn vì có check tồn tại trên GCS/local rồi skip.

6) GOOGLE_APPLICATION_CREDENTIALS trỏ file không tồn tại

  • Python script sẽ báo lỗi auth/credentials file not found.
  • gcloud command có thể báo trạng thái khác nếu CLI login còn nhưng ADC không hợp lệ.
  • Verify nhanh: gcloud auth application-default print-access-token.

7) Thay đổi schema file Parquet trên GCS rồi query External Table cũ

  • Lỗi thường gặp: schema mismatch hoặc cột mới trả về NULL khi query.
  • Tại sao: External Table giữ schema tại thời điểm CREATE, không tự auto-refresh schema mới từ source.
  • Fix: DROP và CREATE OR REPLACE EXTERNAL TABLE để refresh schema metadata.
  • Production lesson: schema evolution cần version control và migration plan rõ (không thay schema ad-hoc trên raw files).
🔥 Debug order đề xuất: (1) đọc full error trong Job details/log, (2) kiểm tra credentials, (3) kiểm tra GCS path, (4) kiểm tra table metadata, (5) kiểm tra bytes scanned/pruning.

2) ERROR HANDLING — Pipeline fail giữa chừng thì sao?

`extras/web_to_gcs.py` hiện gọi requests.get() rồi ghi file ngay, không có raise_for_status() và không retry. Nếu HTTP trả về 404/500, bạn có thể ghi nhầm error payload thành file dữ liệu rồi upload lên GCS.

  • Fix 1: fail-fast bằng raise_for_status() sau mỗi request.
  • Fix 2: retry có chủ đích với exponential backoff để chịu lỗi mạng tạm thời.
  • Fix 3: dead-letter strategy: file lỗi/metadata lỗi đưa vào bucket/path riêng để audit và reprocess.
● ● ●retry_download.py
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=30))
def download_csv(url, local_path):
    r = requests.get(url, timeout=60)
    r.raise_for_status()

3) COST CONTROL — Query tốn bao nhiêu và chặn vượt ngân sách thế nào?

Với BigQuery on-demand, bạn trả tiền theo bytes billed. Bytes processed là dữ liệu engine cần đọc, còn bytes billed là lượng tính phí sau chính sách pricing/rounding. Senior mindset là estimate trước, chặn trần cost, và tối ưu retention.

  • Dry run: estimate bytes trước khi query thật bằng API (dry_run=True).
  • Cap per query: đặt maximum_bytes_billed để query fail nếu vượt giới hạn.
  • Storage hygiene: đặt partition expiration để tự dọn data cũ.
● ● ●bq_cost_guardrail.py
from google.cloud import bigquery

client = bigquery.Client()
query = """
SELECT COUNT(*)
FROM `taxi-rides-ny.nytaxi.yellow_tripdata_partitioned`
WHERE DATE(tpep_pickup_datetime) BETWEEN '2019-06-01' AND '2019-06-30'
"""

cfg = bigquery.QueryJobConfig(dry_run=True, use_query_cache=False)
dry_job = client.query(query, job_config=cfg)
print(f"Estimated bytes: {dry_job.total_bytes_processed:,}")

run_cfg = bigquery.QueryJobConfig(maximum_bytes_billed=2_000_000_000)
run_job = client.query(query, job_config=run_cfg)
run_job.result()

4) SECURITY — Setup hiện tại có rủi ro gì?

`.env-example` dùng GOOGLE_APPLICATION_CREDENTIALS=Path/to/key.json. JSON key trên disk rất tiện cho local lab nhưng rủi ro cao nếu lộ file hoặc commit nhầm git: attacker có thể truy cập trực tiếp tài nguyên GCP theo quyền service account đó.

  • Không giữ key file lâu dài trên CI: ưu tiên Workload Identity Federation cho GitHub Actions/GCE/GKE.
  • Least privilege: chỉ cấp đúng role cần thiết như roles/storage.objectViewer, roles/bigquery.dataEditor.

🧠 Phần 1 — BigQuery internals (Senior must-know) ▼

Capacitor: columnar format của BigQuery

BigQuery dùng storage engine dạng cột (Capacitor) để chỉ đọc cột cần thiết, nén tốt và phục vụ scan lớn. Bạn không thao tác trực tiếp file layout như Parquet/ORC trong object storage vì BigQuery abstraction hóa tầng lưu trữ này để tối ưu nội bộ.

Execution tree (Dremel): root → mixer → leaf

Query được chia thành stages chạy song song: leaf đọc dữ liệu, mixer tổng hợp trung gian, root trả kết quả cuối. Chỗ tốn nhất thường nằm ở shuffle/repartition giữa stages, đặc biệt khi join/group by trên cardinality cao.

Slot: compute unit và hiểu lầm phổ biến

Slot là đơn vị tài nguyên compute mà query tiêu thụ. Nhiều slot hơn không luôn nhanh hơn nếu bottleneck nằm ở data skew, shuffle lớn, hoặc SQL anti-pattern. Quan trọng là tối ưu plan trước khi mua thêm compute.

Join strategies + anti-patterns gây slot contention

  • Broadcast join: table nhỏ được gửi đến worker xử lý table lớn (thường nhanh hơn khi dim nhỏ).
  • Shuffle join: cả hai phía phải repartition theo key, tốn network/shuffle nhiều hơn.
  • Anti-pattern: cross join vô ý, DISTINCT trên bảng cực lớn không filter, correlated subquery nặng.
⚠️ Rule of thumb: giảm dữ liệu trước join (filter/aggregate sớm), hạn chế `SELECT *`, và đọc execution details thay vì đoán.
🤖 Phần 2 — BigQuery ML production (Senior must-know) ▼

Training-serving skew và data drift

`big_query_ml.sql` train trên historical table `yellow_tripdata_ml`. Khi serving thực tế, feature pipeline có thể khác logic transform lúc train, gây skew và giảm chất lượng dự đoán.

Model versioning không làm vỡ production

  • Blue/Green: deploy model mới song song model cũ, chuyển traffic dần.
  • Champion/Challenger: model mới chạy shadow, so metric trước khi promote.

BQML không tự retrain

BQML mạnh ở SQL-first ML nhưng không tự orchestration lifecycle. Scheduled retraining cần Cloud Scheduler/Kestra/Airflow/Vertex AI Pipelines để trigger theo lịch hoặc theo drift signal.

Serving trade-off: TF Serving vs Vertex AI Endpoint

`extract_model.md` mô tả flow export model từ BigQuery rồi serve bằng TensorFlow Serving. Cách này linh hoạt và rẻ ở quy mô nhỏ, nhưng production lớn thường cần managed endpoint (autoscale, IAM, monitoring) như Vertex AI Endpoint.

🏗️ Phần 3 — Data modeling trong DWH (Senior must-know) ▼

Star schema vs wide denormalized table

Star schema (fact + dimensions) giúp rõ semantics và governance; wide table giảm join runtime cho BI query thường dùng. BigQuery thường xử lý tốt wide table có partition/cluster hợp lý, nhưng governance/lineage có thể khó hơn nếu mọi thứ dồn vào một bảng khổng lồ.

SCD Type 1 vs Type 2

  • Type 1 (overwrite): giữ trạng thái mới nhất, mất history; hợp cho dữ liệu sửa lỗi không cần truy vết.
  • Type 2 (history): thêm dòng mới với valid_from/valid_to/is_current; hợp cho phân tích theo thời điểm.

Partition strategy cho fact table

  • Partition theo event date: hợp khi business query theo thời điểm sự kiện.
  • Partition theo ingestion date: hợp khi event time không sạch hoặc cần operational observability.

Late-arriving data xử lý thế nào?

Record "hôm qua mới đến hôm nay" vẫn nên ghi vào partition của event date nếu mục tiêu là báo cáo business đúng lịch sử. Điều này kéo theo nhu cầu re-open partition cũ và incremental MERGE thay vì append mù.

Junior 1) Data Warehouse khác Database thông thường ở điểm nào cơ bản nhất?▼

⚠️ Red flag: Đừng nhầm rằng query nhanh là luôn rẻ.

⚠️ Red flag: Đừng nghĩ cluster thay thế hoàn toàn partition.

⚠️ Red flag: Đừng trả lời "chạy lại toàn bộ cho chắc" khi data đã lớn.

⚠️ Red flag: Đừng chỉ nói "Parquet hiện đại hơn" mà không nêu cost/perf.
