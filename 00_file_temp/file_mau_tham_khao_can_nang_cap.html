2. Tại Sao Dimensional Modeling Là Nền Tảng? ▼

Dimensional Modeling theo Kimball là trọng tâm của homework trong intermediate-bootcamp/materials/1-dimensional-data-modeling/homework/homework.md: thiết kế actors dạng cumulative, rồi xây actors_history_scd Type 2 bằng backfill + incremental. Nếu làm đúng luồng này, bạn nắm được nền tảng modeling production thay vì chỉ học định nghĩa.

Kimball 4-step (áp dụng trực tiếp cho tuần này)

  • Bước 1 — Chọn business process: theo repo là theo dõi lịch sử actor theo năm từ actor_films.
  • Bước 2 — Declare grain: mỗi record trong actors là 1 actor tại 1 năm; mỗi record trong actors_history_scd là 1 khoảng thời gian hiệu lực.
  • Bước 3 — Identify dimensions: actor, quality_class, is_active, thời gian hiệu lực (start/end).
  • Bước 4 — Identify facts/measures: films array, avg rating gần nhất để phân lớp star/good/average/bad, và trạng thái active/inactive.

❌ Không biết Dimensional Modeling → Những vấn đề này

  • Thiết kế schema theo kiểu OLTP cho analytics → query chậm, JOIN phức tạp không cần thiết
  • Không biết SCD → lịch sử dimension bị mất khi update, audit trail không có
  • Không định nghĩa grain rõ → fact table có duplicate row không ai phát hiện
  • Không biết Cumulative Table → phải scan toàn bộ event history mỗi lần query

✅ Sau Tuần 5–6 bạn có thể

  • Định nghĩa grain cho bất kỳ fact table nào và giải thích trade-off
  • Implement SCD Type 2 hoàn chỉnh với backfill + incremental SQL
  • Thiết kế Star Schema cho e-commerce / streaming / fintech domain
  • Xây Cumulative Table hiệu quả thay vì scan toàn bộ event log
  • Giải thích design decisions trong interview (tại sao chọn SCD2 thay SCD1)

1. Menu Tổng Thể — Grain, Fact & Dimension ▼

Grain = định nghĩa chính xác "1 row trong fact table đại diện cho cái gì". Đây là quyết định quan trọng nhất trong data modeling — sai grain = toàn bộ schema sai.

📌 Mục đích module: Sau phần này bạn phải tự mô tả được grain, tách fact/dimension đúng, và giải thích vì sao chọn surrogate key hay natural key.

Ví dụ định nghĩa Grain

DomainGrain (đúng)Grain (sai — quá thô)
E-commerce1 row = 1 order line item (1 sản phẩm trong 1 đơn hàng)1 row = 1 đơn hàng (mất chi tiết sản phẩm)
Streaming1 row = 1 lần user xem 1 content trong 1 session1 row = 1 user trong 1 ngày
Fintech1 row = 1 transaction giữa 2 account1 row = tổng tiền user trong 1 ngày
HR1 row = 1 employee trong 1 pay period1 row = 1 employee (snapshot)
⚠️ Nguyên tắc: Grain càng atomic (nhỏ nhất có thể) thì càng linh hoạt cho analytics. Có thể aggregate lên, nhưng không thể drill down xuống khi đã aggregate.

✅ Rule thực chiến: giữ natural key như business key để trace lineage, nhưng join fact-dimension bằng surrogate key để bảo toàn lịch sử SCD2.

ddl_fact_dim.sql — PostgreSQL
-- DIMENSION TABLE: dim_customer
-- Surrogate key + natural key pattern
CREATE TABLE dims.dim_customer (
    customer_key    BIGSERIAL PRIMARY KEY,          -- surrogate key
    customer_id     TEXT NOT NULL,                   -- natural key từ source
    full_name       TEXT NOT NULL,
    email           TEXT,
    country         TEXT,
    segment         TEXT,                             -- 'premium', 'standard'
    is_active       BOOLEAN DEFAULT TRUE,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- FACT TABLE: fact_orders
-- Grain: 1 row = 1 order line item
CREATE TABLE facts.fact_orders (
    order_line_key  BIGSERIAL PRIMARY KEY,
    order_id        TEXT NOT NULL,
    line_number     INTEGER NOT NULL,
    customer_key    BIGINT NOT NULL REFERENCES dims.dim_customer(customer_key),
    product_key     BIGINT NOT NULL,
    date_key        INTEGER NOT NULL,                 -- YYYYMMDD integer
    -- Measures (additive facts)
    quantity        INTEGER NOT NULL,
    unit_price      NUMERIC(12,2) NOT NULL,
    discount_amount NUMERIC(12,2) DEFAULT 0,
    net_amount      NUMERIC(12,2) GENERATED ALWAYS AS
                    (quantity * unit_price - discount_amount) STORED,
    order_date      DATE NOT NULL
);

-- Index trên FK để join nhanh
CREATE INDEX idx_fact_orders_customer ON facts.fact_orders(customer_key);
CREATE INDEX idx_fact_orders_date     ON facts.fact_orders(date_key);

A. Database báo lỗi vì không thể lưu 5 sản phẩm trong 1 row
B. Mất chi tiết sản phẩm — không thể phân tích revenue theo SKU, không biết sản phẩm nào bán trong đơn
C. Query trả về lỗi khi GROUP BY product
D. Tạo ra duplicate rows cho mỗi sản phẩm
Tại sao Fact table dùng Surrogate Key thay vì Natural Key từ source?
A. BIGSERIAL nhanh hơn TEXT khi join
B. Ổn định khi source rekey, hỗ trợ SCD Type 2 (nhiều version của cùng 1 entity), tránh phụ thuộc source schema
C. Surrogate key tự động mã hóa data nhạy cảm
D. Natural key không thể là PRIMARY KEY trong PostgreSQL
Periodic Snapshot fact table có điểm gì khác biệt so với Transaction fact?
A. Periodic Snapshot không có measure nào, chỉ có dimension keys
B. Chụp trạng thái tại điểm thời gian cố định; measure semi-additive — không SUM được theo time
C. Periodic Snapshot nhỏ hơn Transaction fact vì bỏ bớt row
D. Periodic Snapshot chỉ dùng cho real-time pipeline
Trong Oracle, tại sao Virtual Column (GENERATED ALWAYS AS ... VIRTUAL) ở fact table tốt hơn lưu giá trị tính sẵn?
A. Virtual column tự động tạo index, giúp query nhanh hơn
B. Không tốn storage — tính on-the-fly, luôn consistent khi input thay đổi, không cần update thêm
C. Tự động mã hóa column khi lưu vào disk
D. Compress tự động để tiết kiệm storage
Một Fact table có measure "account_balance". Tại sao không nên SUM balance của tất cả ngày trong tháng?
A. Sẽ gây lỗi database vì balance là NUMERIC
B. Balance là semi-additive — SUM theo time vô nghĩa (đếm balance 30 lần). Dùng LAST_VALUE hoặc AVG theo time
C. Sẽ overflow vì balance quá lớn khi cộng 30 ngày
D. Cú pháp SQL không cho phép SUM cột balance
date_key INTEGER dạng YYYYMMDD (ví dụ 20240115) trong Fact table có lợi ích gì so với DATE type?
A. Tự động convert timezone khi query
B. JOIN nhanh hơn, partition efficient, dễ link dim_date để lấy quarter/fiscal_year/holiday attributes
C. Tốn ít storage hơn DATE type
D. Tự động validate ngày hợp lệ (không cho phép 20240231)
Accumulating Snapshot fact table phù hợp nhất cho loại analysis nào?
A. Daily revenue trend theo thời gian
B. Phân tích lifecycle/fulfillment — track order qua nhiều milestone (placed→shipped→delivered)
C. Inventory level snapshot cuối mỗi ngày
D. Tổng số active customer theo tháng
Trong PostgreSQL, BIGSERIAL khác gì với BIGINT GENERATED ALWAYS AS IDENTITY?
A. BIGSERIAL chỉ dùng được trong PostgreSQL 10+, IDENTITY dùng được từ phiên bản cũ hơn
B. BIGSERIAL là shorthand PostgreSQL-specific, cho phép override; IDENTITY là SQL standard, strict hơn và là cách hiện đại
C. BIGSERIAL tạo integer; IDENTITY tạo UUID
D. IDENTITY không thể dùng làm PRIMARY KEY
Khi nào một Dimension attribute nên nằm trong Fact table thay vì Dimension table?
A. Khi attribute là thông tin khách hàng (tên, email)
B. Khi là Degenerate Dimension — attribute từ source transaction (order_number) không có thêm attributes để tạo dim riêng
C. Khi attribute có cardinality cao (nhiều giá trị unique)
D. Khi attribute là date — luôn để trong fact để tránh join thêm
INDEX trên Foreign Key của Fact table (ví dụ customer_key) có tác dụng gì trong analytics query?
A. Tăng tốc INSERT vào fact table
B. Tăng tốc JOIN với dimension table, đặc biệt khi filter theo dimension attribute (nested loop join thay vì hash join)
C. Đảm bảo referential integrity giữa fact và dim
D. Prevent duplicate fact rows theo customer

    quality_class   TEXT NOT NULL,
    is_active       BOOLEAN NOT NULL,
    start_date      INTEGER NOT NULL,  -- year bắt đầu hiệu lực
    end_date        INTEGER,           -- NULL hoặc max year, tùy convention
    is_current      BOOLEAN DEFAULT TRUE,
    current_year    INTEGER,           -- tùy chọn, giống pattern lecture-lab
    PRIMARY KEY (actor_id, start_date)

-- BACKFILL QUERY: populate toàn bộ lịch sử 1 lần
INSERT INTO public.actors_history_scd
WITH actor_changes AS (
    SELECT
        actor_id, actor, quality_class, is_active, current_year,
        LAG(quality_class) OVER (PARTITION BY actor_id ORDER BY current_year)
            AS prev_quality,
        LAG(is_active)     OVER (PARTITION BY actor_id ORDER BY current_year)
            AS prev_active
    FROM public.actors
),
change_flags AS (
    SELECT *,
        CASE WHEN quality_class != prev_quality
               OR is_active    != prev_active
               OR prev_quality IS NULL   -- first record
             THEN 1 ELSE 0
        END AS is_change
    FROM actor_changes
),
groups AS (
    SELECT *,
        SUM(is_change) OVER (PARTITION BY actor_id ORDER BY current_year)
            AS change_group
    FROM change_flags
)
SELECT
    actor_id,
    actor,
    quality_class,
    is_active,
    MIN(current_year)                    AS start_date,
    MAX(current_year)                    AS end_date,
    MAX(current_year) = (SELECT MAX(current_year) FROM public.actors)
                                         AS is_current
FROM groups
GROUP BY actor_id, actor, quality_class, is_active, change_group
ORDER BY actor_id, start_date;

-- INCREMENTAL: thêm 1 năm mới vào actors_history_scd
WITH last_year_scd AS (
    -- Lấy record current của năm trước
    SELECT actor_id, actor, quality_class, is_active, start_date
    FROM public.actors_history_scd
    WHERE is_current = TRUE
),
this_year AS (
    -- Data năm hiện tại từ source
    SELECT actor_id, actor, quality_class, is_active, current_year
    FROM public.actors
    WHERE current_year = :target_year
),
changed AS (
    -- Actor thay đổi quality_class hoặc is_active
    SELECT ty.*
    FROM this_year ty
    JOIN last_year_scd ly USING (actor_id)
    WHERE ty.quality_class != ly.quality_class
       OR ty.is_active    != ly.is_active
)
-- Step 1: Expire old records cho actor bị thay đổi
UPDATE public.actors_history_scd
SET end_date   = :target_year - 1,
    is_current = FALSE
WHERE actor_id IN (SELECT actor_id FROM changed)
  AND is_current = TRUE;

-- Step 2: Insert new records
INSERT INTO public.actors_history_scd
    (actor_id, actor, quality_class, is_active, start_date, end_date, is_current)
SELECT
    actor_id, actor, quality_class, is_active,
    :target_year,   -- start_date = năm mới
    NULL,           -- end_date = NULL (current)
    TRUE
FROM changed;

-- BACKFILL: Oracle dùng LAG window function (tương tự PostgreSQL)
INSERT INTO actors_history_scd
    (actor_id, actor, quality_class, is_active, start_date, end_date, is_current)
WITH actor_changes AS (
    SELECT
        actor_id, actor, quality_class, is_active, current_year,
        LAG(quality_class) OVER (PARTITION BY actor_id ORDER BY current_year)
            AS prev_quality,
        LAG(is_active)     OVER (PARTITION BY actor_id ORDER BY current_year)
            AS prev_active
    FROM actors
),
change_flags AS (
    SELECT actor_id, actor, quality_class, is_active, current_year,
        CASE WHEN quality_class != prev_quality
               OR is_active    != prev_active
               OR prev_quality IS NULL
             THEN 1 ELSE 0
        END AS is_change
    FROM actor_changes
),
groups AS (
    SELECT actor_id, actor, quality_class, is_active, current_year,
        SUM(is_change) OVER (PARTITION BY actor_id ORDER BY current_year)
            AS change_group
    FROM change_flags
)
SELECT
    actor_id,
    actor,
    quality_class,
    is_active,
    MIN(current_year)  AS start_date,
    MAX(current_year)  AS end_date,
    CASE WHEN MAX(current_year) = (SELECT MAX(current_year) FROM actors)
         THEN 1 ELSE 0
    END AS is_current
FROM groups
GROUP BY actor_id, actor, quality_class, is_active, change_group
ORDER BY actor_id, MIN(current_year);

    END                                   AS films,
    :year                                 AS current_year,
    -- quality_class based on avg rating of recent films
    CASE WHEN AVG(t.rating) > 8    THEN 'star'
         WHEN AVG(t.rating) > 7    THEN 'good'
         WHEN AVG(t.rating) > 6    THEN 'average'
         ELSE 'bad'
    END                                   AS quality_class,
    t.actor_id IS NOT NULL              AS is_active
FROM yesterday y
FULL OUTER JOIN today_films t USING (actor_id)
GROUP BY COALESCE(t.actor_id, y.actor_id), y.films, t.actor_id;

        LAG(is_active)     OVER (PARTITION BY actor_id ORDER BY year) AS prev_ia,
        SUM(CASE WHEN quality_class != LAG(quality_class) OVER

                        (PARTITION BY actor_id ORDER BY year)

    FROM actors

    SELECT min_yr + (LEVEL - 1) AS yr
    FROM year_range
    CONNECT BY min_yr + (LEVEL - 1) <= max_yr

    FROM (SELECT DISTINCT actor_id FROM actors) a

                       OR is_active != LAG(is_active) OVER
                        (PARTITION BY actor_id ORDER BY year)
                       OR LAG(quality_class) OVER
                        (PARTITION BY actor_id ORDER BY year) IS NULL
                  THEN 1 ELSE 0 END)

        WHEN t.avg_rating > 8 THEN 'star'
        WHEN t.avg_rating > 7 THEN 'good'
        WHEN t.avg_rating > 6 THEN 'average'
        ELSE 'bad'
    END                               AS quality_class,
    CASE WHEN t.actor_id IS NOT NULL THEN 1 ELSE 0 END AS is_active,
    :target_year                         AS current_year
FROM yesterday y
FULL OUTER JOIN today t
    ON y.actor_id = t.actor_id;
COMMIT;

📝 Yêu cầu

Sau khi backfill và incremental cho actors_history_scd, viết query validation để kiểm tra: không overlap interval, mỗi actor có đúng 1 current record, incremental không làm giảm dữ liệu lịch sử.

scd_validation.sql
-- 1) Không overlap interval cho cùng actor
SELECT a.actor_id, a.start_date, a.end_date, b.start_date, b.end_date
FROM actors_history_scd a
JOIN actors_history_scd b
  ON a.actor_id = b.actor_id
 AND a.start_date <= COALESCE(b.end_date, 9999)
 AND b.start_date <= COALESCE(a.end_date, 9999)
 AND (a.start_date, COALESCE(a.end_date, 9999))
    <> (b.start_date, COALESCE(b.end_date, 9999));

-- 2) Mỗi actor đúng 1 current record
SELECT actor_id,
       SUM(CASE WHEN is_current THEN 1 ELSE 0 END) AS current_rows
FROM actors_history_scd
GROUP BY actor_id
HAVING SUM(CASE WHEN is_current THEN 1 ELSE 0 END) != 1;
