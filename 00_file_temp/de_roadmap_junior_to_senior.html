01 · Tổng quan

Bức tranh toàn cảnh

Trước khi bắt đầu, bạn cần hiểu DE làm gì, hệ sinh thái trông như thế nào, và tại sao các công nghệ trong Zoomcamp lại được chọn.

Data Engineer làm gì?

Xây dựng pipeline

Di chuyển dữ liệu từ nguồn (API, DB, file) vào hệ thống lưu trữ, đảm bảo đúng, đủ, kịp thời.

Quản lý data platform

Infra, scheduling, monitoring, cost — đảm bảo toàn bộ hệ thống data chạy ổn định.

Đảm bảo data quality

Validate schema, detect anomalies, handle late/duplicate data, build trust với stakeholders.

Enable analytics

Biến raw data thành dạng mà analyst và scientist có thể dùng được — modeling, transformation.

03 · Capstone xuyên suốt

Project xuyên suốt 7 module

Dùng một project end-to-end duy nhất để kết nối toàn bộ Zoomcamp: từ setup hạ tầng đến batch + streaming và lớp mart cho BI.

Đề bài gợi ý & 3 Phase triển khai

Build nền tảng phân tích chuyến taxi: ingest dữ liệu trip + weather + event clickstream, xây dashboard demand theo giờ/khu vực, đồng thời có cảnh báo near-real-time khi surge tăng bất thường.

Phase 1: MVP (Minimum Viable Product)

Taxi batch pipeline → BigQuery → dbt → Dashboard cơ bản.

Phase 2: Mở rộng & Đảm bảo chất lượng

Thêm Weather API + Data quality checks (Great Expectations/dbt tests).

Phase 3: Real-time & Alerting

Thêm Kafka/Flink streaming clickstream + Alerting.
Cảnh báo: Không bắt đầu streaming/Flink nếu MVP batch ở Phase 1 chưa chạy end-to-end trơn tru.

Data flow diagram (text)
Sources
  ├─ NYC taxi trips (batch files/API)
  ├─ Weather API (batch)
  └─ App events (stream)

Ingestion Layer
  ├─ Module 01: Docker services + Terraform infra
  └─ Raw landing zone (GCS/Bucket + staging tables)

Orchestration Layer
  └─ Module 02: Kestra schedules ingestion + retries + alerts

Warehouse Layer
  └─ Module 03: BigQuery raw tables (partitioned)

Transformation Layer
  └─ Module 04: dbt (staging → intermediate → marts)

Platform Layer
  └─ Module 05: Asset lineage, ownership, SLA, quality checks

Batch Compute
  └─ Module 06: Spark job for heavy aggregations/backfills

Streaming Compute
  └─ Module 07: Kafka + Flink for near-real-time metrics

Serving
  ├─ BI dashboard (hourly/daily KPIs)
  └─ Alert channel when anomaly detected
Milestone theo từng module
  • Module 01: Container hóa local stack, Terraform tạo bucket + dataset + service account.
  • Module 02: Orchestrate job ingest hàng ngày, có retry/backoff, có failure notification.
  • Module 03: Nạp raw vào BigQuery, partition/cluster chuẩn, kiểm soát schema drift.
  • Module 04: Build dbt models theo staging/intermediate/marts + tests + docs.
  • Module 05: Quản lý asset/data contract, định nghĩa owner, SLA, runbook.
  • Module 06: Spark batch để backfill và aggregate dataset lớn (cost/perf tối ưu).
  • Module 07: Kafka/Flink streaming pipeline cho chỉ số realtime + checkpointing.

Output cuối cùng

1 repo chạy được end-to-end, diagram kiến trúc, dashboard KPI, tài liệu runbook, và video demo ngắn 5–7 phút.

Cách kể trong phỏng vấn

Dùng format: Problem → Design → Failure đã gặp → Trade-off đã chọn → Kết quả đo được (latency/cost/reliability).

Final Project bắt buộc phải có
  • Terraform (IaC) + Docker (runtime consistency).
  • Orchestration pipeline + ingestion layer (batch/API/stream source).
  • BigQuery warehouse + dbt transformation (staging/intermediate/marts).
  • Spark batch job cho workload lớn hoặc backfill nặng.
  • Kafka/Flink streaming flow cho realtime metrics/use case.
  • Dashboard phục vụ stakeholder (Looker Studio/Metabase/BI tool tương đương).
  • CI/CD pipeline cho code + Terraform + dbt checks.
  • Alerting (pipeline fail, freshness, lag, cost spike) + on-call runbook.
  • Testing: unit/integration/data quality tests với quality gates.
  • Documentation: architecture, lineage, data contract, handover guide.
  • Cost estimation: baseline + monthly projection + optimization notes.
  • Failure recovery plan: retry/replay/backfill/rollback theo incident type.
04 · Debug discipline

Run / Break / Fix Logbook

Mỗi buổi học phải để lại logbook. Đây là bằng chứng năng lực debug và tư duy production, không phải checklist hình thức.

Template cho mỗi lần học
Tôi chạy command gì?Ghi chính xác command + input dataset/config đã dùng.
Kết quả mong đợi?Nêu rõ expected output: table nào tạo ra, job state nào thành công, metric nào tăng/giảm.
Lỗi xảy ra?Paste nguyên văn error quan trọng, không paraphrase.
Tôi đọc log ở đâu?Container logs, orchestrator logs, Spark UI, BigQuery job history, Kafka consumer lag...
Root cause là gì?Phân biệt symptom vs root cause; nêu bằng chứng xác minh.
Fix thế nào?Fix cụ thể ở code/config + vì sao fix này giải quyết đúng root cause.
Nếu lỗi này xảy ra production lúc 3h sáng?Escalation path, rollback/backfill plan, alert, owner chịu trách nhiệm.
Tôi học được gì?1–2 nguyên tắc rút ra để tránh lặp lại ở module sau.
Rule: Không được đóng buổi học nếu chưa có ít nhất 1 entry logbook có root cause rõ ràng.
05 · Workshop chuyên sâu

Workshop: Data Ingestion chuyên sâu

Từ mức “ingest được” lên mức production-ready: kiểm soát tốc độ gọi API, incremental strategy chuẩn, và contract để tránh vỡ downstream.

WS1 dlt + API ingestion patterns ›
Pro-tier patterns
  • dlt: Chuẩn hóa extraction/load pipeline, schema evolution có kiểm soát.
  • API pagination: Cursor-based/page-based pagination có checkpoint để resume.
  • Rate limiting: Token bucket / delay theo quota provider để tránh 429.
  • Retry/backoff: Exponential backoff + jitter, phân loại retryable vs non-retryable errors.
  • CDC vs timestamp incremental: Chọn theo độ tin cậy nguồn và yêu cầu audit/replay.
  • Data contract: Định nghĩa schema bắt buộc + nullability + semantic rules ngay từ ingestion.
Lab phải làm
  • Build 1 pipeline dlt ingest API phân trang có resume sau khi fail giữa chừng.
  • Giả lập 429/5xx và chứng minh retry + backoff hoạt động đúng.
  • So sánh 2 chiến lược incremental (CDC và updated_at) trên cùng dataset.
  • Viết data contract và chặn pipeline khi payload vi phạm contract.
06 · Senior thinking by module

Senior Questions trong từng module

Mỗi module đều phải tự trả lời đủ 5 câu hỏi dưới đây với ngữ cảnh cụ thể của module đó.

M01 Docker + Terraform ›
Pro-tier patterns
  • Multi-stage Docker build để giảm image size và bề mặt tấn công.
  • Secrets management: không hardcode; dùng secret manager hoặc injected runtime variables.
  • Remote Terraform state (GCS/S3) + state locking để tránh race conditions.
  • Tách Terraform theo modules, variables, locals để tái sử dụng và dễ review.
  • CI/CD cho IaC: tự động terraform fmt/validate/plan, apply có approval.
Senior Questions
  • Nếu provisioning fail giữa chừng thì rollback state/cleanup resource thế nào?
  • Nếu số service và môi trường tăng x100 thì bottleneck nằm ở image build, state lock hay networking?
  • Nếu chi phí cloud tăng mạnh thì tối ưu resource sizing, lifecycle policy và môi trường dev ra sao?
  • Nếu schema nguồn đổi, hạ tầng ingest cần versioning/contract thế nào để không vỡ downstream?
  • Nếu bàn giao cho Junior, cần document bootstrap, secrets handling, state backend, runbook gì?
M02 Workflow Orchestration ›
Pro-tier patterns
  • Idempotency cho từng task để rerun an toàn, không tạo duplicate.
  • Backfill + catchup strategy theo partition/time window, có giới hạn concurrency.
  • Dynamic DAG/flow sinh từ config để scale nhiều data source.
  • SLA monitoring cho duration/freshness; alerting theo severity (Slack/Email/Pager).
  • Retry policy tách bạch transient vs permanent failures.
Senior Questions
  • Nếu DAG fail giữa chừng, rerun từ task nào để an toàn dữ liệu?
  • Nếu số lượng workflow x100, bottleneck ở scheduler, queue hay worker concurrency?
  • Nếu cost tăng, nên tối ưu tần suất schedule, retries, compute profile thế nào?
  • Nếu source schema đổi, task validate và branching chiến lược ra sao?
  • Nếu bàn giao Junior, cần document dependency graph, SLA, on-call flow gì?
M03 Data Warehouse (BigQuery) ›
Pro-tier patterns
  • BigQuery cost optimization: partition filters bắt buộc, tránh SELECT *, theo dõi bytes scanned.
  • Kimball modeling: fact/dimension rõ grain, conformed dimensions cho nhiều mart.
  • RLS/CLS để bảo vệ dữ liệu nhạy cảm theo role và domain.
  • Materialized views cho dashboard/query lặp lại, cân bằng freshness vs cost.
  • Reservation/slot strategy theo workload nếu team query đồng thời lớn.
Senior Questions
  • Nếu load fail giữa chừng thì đảm bảo atomicity/idempotency khi nạp lại thế nào?
  • Nếu volume x100, bottleneck ở partition design, join pattern hay concurrent slots?
  • Nếu bill tăng mạnh, tối ưu bytes scanned, materialization và retention ra sao?
  • Nếu source schema đổi, migration và compatibility strategy cho tables downstream thế nào?
  • Nếu bàn giao Junior, cần document naming convention, partition policy, cost guardrail gì?
M04 dbt Analytics Engineering ›
Pro-tier patterns
  • dbt Slim CI: chỉ run/test model bị ảnh hưởng trong Pull Request.
  • Incremental merge vs append: chọn theo business key, late updates và lịch sử dữ liệu.
  • dbt macros + packages để chuẩn hóa logic SQL tái sử dụng.
  • Data quality gates: fail pipeline khi test critical thất bại.
  • State-based selection để giảm thời gian CI/CD và chi phí query.
Senior Questions
  • Nếu run fail giữa chừng, chọn chạy lại model nào và state selection ra sao?
  • Nếu data x100, bottleneck ở incremental logic, tests hay model dependency nào?
  • Nếu cost tăng, tối ưu materialization (view/table/incremental) thế nào?
  • Nếu schema nguồn đổi, update staging contract và test coverage ra sao?
  • Nếu bàn giao Junior, document lineage, model contract, và release checklist gì?
M05 Data Platform / Asset Thinking ›
Pro-tier patterns
  • Modern data stack evaluation theo use case: managed services vs self-hosted components.
  • Metadata management: ownership, SLA/SLO, freshness, classification.
  • Data lineage end-to-end từ source đến dashboard để tăng tốc RCA.
  • Asset-based platform thinking: pipeline như product có contract và lifecycle.
  • Governance workflow cho change management và incident review.
Senior Questions
  • Nếu một asset fail, ảnh hưởng dây chuyền tới asset nào và degrade mode là gì?
  • Nếu asset count x100, bottleneck ở metadata, ownership hay observability?
  • Nếu cost platform tăng, chuẩn hóa shared services và chargeback thế nào?
  • Nếu schema đổi liên tục, quản trị data contract và versioning sao cho bền?
  • Nếu bàn giao Junior, cần document ownership matrix, SLA/SLO, incident template gì?
M06 Batch Processing (Spark) ›
Pro-tier patterns
  • Spark memory tuning: executor/core/memory overhead theo workload.
  • Spill to disk awareness: theo dõi shuffle spill và tối ưu partition sizing.
  • Data skew mitigation: salting keys, repartition có chủ đích, skew hints.
  • Broadcast join + AQE để giảm shuffle và tự điều chỉnh plan runtime.
  • Spark unit testing (Pytest/chispa) cho transformation logic quan trọng.
Senior Questions
  • Nếu job fail giữa chừng, checkpoint/output commit strategy nào giúp rerun an toàn?
  • Nếu data x100, bottleneck ở shuffle, skew, IO hay executor memory?
  • Nếu cost compute tăng, tối ưu partitioning, autoscaling, storage format ra sao?
  • Nếu schema thay đổi, xử lý evolution của parquet/table và compatibility thế nào?
  • Nếu bàn giao Junior, document Spark UI reading guide và tuning playbook gì?
M07 Streaming (Kafka + Flink) ›
Pro-tier patterns
  • Exactly-once semantics theo phạm vi end-to-end, không chỉ producer side.
  • Partition strategy: key design, consumer scaling, rebalance impact.
  • Consumer lag monitoring + auto-remediation/playbook khi lag vượt ngưỡng.
  • Schema evolution với Avro compatibility (backward/forward/full) qua registry.
  • Replay và DLQ strategy để khôi phục dữ liệu khi downstream fail.
Senior Questions
  • Nếu stream job fail giữa chừng, offset/checkpoint recovery bảo toàn dữ liệu thế nào?
  • Nếu throughput x100, bottleneck ở broker partitions, consumer lag hay state backend?
  • Nếu cost tăng, tối ưu retention, serialization format, window strategy ra sao?
  • Nếu source schema đổi, áp dụng schema registry và compatibility mode thế nào?
  • Nếu bàn giao Junior, document replay strategy, DLQ flow, on-call runbook gì?
07 · Giai đoạn học

Từng bước — lý thuyết đến thực chiến

Mỗi module đều đi qua đúng 5 bước này. Đừng nhảy bước.

01 Lý thuyết nền — hiểu BỨC TRANH trước 2–3 ngày / module ›
Câu hỏi cần trả lời được trước khi code
  • Công nghệ này giải quyết vấn đề gì? Vấn đề đó xuất hiện từ khi nào?
  • Trước khi có nó, người ta làm thế nào? Tại sao cách cũ không đủ?
  • Nó hoạt động như thế nào ở mức tổng quát (không cần biết implementation detail)?
  • Khi nào NÊN dùng? Khi nào KHÔNG nên dùng?
  • Các alternatives là gì? Trade-off giữa chúng?
Ví dụ — Module 01: Docker
Vấn đề Docker giải quyết: "Works on my machine" — code chạy được trên laptop dev nhưng fail trên server production vì khác OS, khác Python version, khác library. Docker đóng gói toàn bộ môi trường vào một container chạy giống hệt nhau ở bất kỳ đâu.
  • Image = bản thiết kế (blueprint) — chứa OS, runtime, dependencies
  • Container = instance đang chạy từ image đó — isolated, ephemeral
  • Volume = cách persist data ra ngoài container (vì container mất data khi stop)
  • Network = cách các container nói chuyện với nhau
  • docker-compose = định nghĩa và chạy nhiều container cùng lúc
Ví dụ — Module 01: Terraform
  • Vấn đề: Click tay trên GCP/AWS console → không reproducible, không version control, dễ sai
  • Terraform giải quyết: Infrastructure as Code — viết file .tf mô tả hạ tầng, Terraform tự tạo/update/destroy
  • State file: Terraform nhớ "hiện tại hạ tầng trông như thế nào" để biết cần thay đổi gì
  • Plan → Apply: Xem trước sẽ thay đổi gì TRƯỚC khi thực sự thực thi
Đừng học syntax trước. Đọc docs, xem video overview, hiểu WHY trước. Syntax học sau 30 phút. WHY mất cả tuần mới ngấm.
02 Đọc code trong repo — hiểu người ta build thế nào 1–2 ngày / module ›
Cách đọc code hiệu quả (không đọc dòng theo dòng)

  • Tìm entry point — file nào chạy đầu tiên?

04 Production thinking — tư duy Senior 1–2 ngày / module ›
Sau khi code chạy được, hỏi những câu này
  • Reliability: Pipeline này fail ở đâu được? Làm sao retry an toàn? Có idempotent không?
  • Scale: Nếu data volume x100, bottleneck ở đâu? Cần thay đổi gì?
  • Observability: Khi fail lúc 3h sáng, tôi biết qua kênh nào? Logs ở đâu? Alert thế nào?
  • Cost: Cái này tốn bao nhiêu? Có cách nào rẻ hơn không mất quality?
  • Security: Credentials được lưu thế nào? Có hardcode secret trong code không?
  • Maintainability: Nếu người khác tiếp nhận code này, mất bao lâu để hiểu?
Ví dụ — Module 01 ingestion pipeline
Vấn đề idempotency: Script ingest_data.py chạy lại có bị duplicate data không? Nếu job fail ở giữa, chạy lại từ đầu có an toàn không? → Đây là câu hỏi Junior thường không nghĩ đến, Senior phải trả lời được trước khi deploy. 
# Câu hỏi cần suy nghĩ:
# 1. Nếu postgres restart → data có mất không? Tại sao?
# 2. Nếu chạy ingest_data.py 2 lần → database có 2x data không?
# 3. Làm sao biết ingestion thành công? Log ở đâu?
# 4. Nếu file CSV có row bị lỗi → toàn bộ fail hay bỏ qua row đó?

08 · Kiến thức Senior

Những gì Zoomcamp không dạy đủ

Đây là những concept phân biệt Junior/Mid với Senior. Không cần học ngay — nhưng phải nắm được trước khi apply Senior role.

A Data reliability engineering ›
Idempotency — khái niệm quan trọng nhất

Một operation là idempotent nếu chạy nhiều lần cho kết quả giống chạy một lần. Mọi pipeline production phải idempotent.

# ❌ KHÔNG idempotent — chạy 2 lần = duplicate
INSERT INTO orders SELECT * FROM raw_orders WHERE date = '2024-01-01'

# ✅ Idempotent — chạy bao nhiêu lần cũng được
INSERT INTO orders
SELECT * FROM raw_orders WHERE date = '2024-01-01'
ON CONFLICT (order_id) DO UPDATE SET updated_at = EXCLUDED.updated_at
Backfill — xử lý historical data
  • Pipeline fail 3 ngày → cần chạy lại 3 ngày đó → gọi là backfill
  • Pipeline phải hỗ trợ execution_date parameter để backfill được
  • Backfill lớn thường chạy parallel → cần cẩn thận về resource contention
Late arriving data
  • Event xảy ra 10h tối nhưng server log về lúc 2h sáng hôm sau → late data
  • Cần phân biệt event time (khi event xảy ra) vs processing time (khi process)
  • Watermark: quyết định "sau bao lâu thì không nhận late data nữa"
  • Ảnh hưởng lớn đến streaming pipeline (Kafka, Flink)
B Data modeling chuyên sâu ›
Medallion Architecture (Bronze / Silver / Gold)
Bronze (Raw Layer)
  → Data ingested as-is, no transformation
  → Immutable, append-only, full history
  → Never delete, never modify

Silver (Cleaned Layer)
  → Dedup, validate, standardize types
  → Partial transformation, still granular
  → Source of truth for data team

Gold (Business Layer)
  → Aggregated, business-ready
  → What analysts query
  → Built from Silver, never from Bronze directly
Slowly Changing Dimensions (SCD)
  • SCD Type 1: Overwrite — đơn giản, mất history. Dùng khi history không quan trọng
  • SCD Type 2: Add row — giữ full history với effective_date. Phổ biến nhất
  • SCD Type 3: Add column — lưu current + previous. Ít dùng
Partitioning strategy
  • Partition by date trong BigQuery → query chỉ scan partition cần thiết → tiết kiệm cost
  • Partition by date trong Spark → parallel processing trên từng partition
  • Over-partitioning (quá nhiều partition nhỏ) → overhead lớn hơn lợi ích
  • Under-partitioning → scan toàn bộ data → slow và tốn tiền
C Observability & Monitoring ›
3 trụ cột observability

Metrics

Số đo: rows processed/min, pipeline duration, error rate, lag. Dùng để alert và trending.

Logs

Structured logs (JSON) cho mỗi event quan trọng. Searchable, filterable. Không log quá nhiều.

Traces

End-to-end journey của một request/record qua hệ thống. Quan trọng khi debug distributed system.

Alerting

Alert khi metric vượt threshold. Alert fatigue nguy hiểm — chỉ alert những thứ cần action ngay.

Data quality checks (phải có trong mọi pipeline)
# Kiểm tra phải có:
- Row count (so với hôm qua, không được giảm quá X%)
- Null rate trên critical columns
- Duplicate check trên primary key
- Value range check (tuổi không thể âm, giá không thể = 0)
- Freshness check (data có update đúng giờ không?)
- Schema drift detection (column mới xuất hiện / mất)
D Performance & Cost optimization ›
BigQuery optimization
# Tránh SELECT * — chỉ select column cần thiết
# BigQuery tính cost theo bytes scanned, không theo rows

# ❌ Tốn tiền
SELECT * FROM `project.dataset.table`

# ✅ Tiết kiệm
SELECT user_id, event_type, created_at
FROM `project.dataset.table`
WHERE DATE(created_at) = '2024-01-01'  -- sử dụng partition
Spark optimization
  • Avoid shuffle khi có thể: Shuffle là operation tốn kém nhất (data move across network)
  • Partition tuning: default 200 partitions sau shuffle — tune theo data size
  • Broadcast join: join bảng nhỏ (< vài GB) → broadcast thay vì shuffle join
  • Predicate pushdown: filter sớm nhất có thể, trước join
  • Cache wisely: cache DataFrame dùng nhiều lần, uncache khi xong
09 · Decision Framework

Decision Making — Khi nào KHÔNG dùng tool

Senior không chọn tool theo trend. Senior chọn theo yêu cầu latency, scale, chi phí, năng lực team và rủi ro vận hành.

Docker

Dùng khi nào: Cần môi trường nhất quán, đóng gói service, deploy đa môi trường.

Không dùng khi nào: Script một lần trên máy local, workload cực nhẹ, không cần isolate.

Trade-off: Reproducible hơn nhưng thêm overhead image/build/network.

Ví dụ production: ETL service nhiều dependency Python → dùng Docker để tránh mismatch giữa dev và prod.

Terraform

Dùng khi nào: Hạ tầng cần version control, review, tái tạo nhiều môi trường.

Không dùng khi nào: POC 1 ngày, 1 resource tạm thời, chưa có vòng đời dài.

Trade-off: Governance tốt hơn nhưng thêm learning curve + state management.

Ví dụ production: Tạo BigQuery dataset/bucket theo chuẩn team, review qua PR trước apply.

Orchestration Tool

Dùng khi nào: Có nhiều task phụ thuộc, lịch chạy định kỳ, cần retry/alert/audit.

Không dùng khi nào: Chỉ 1 script chạy thủ công vài lần/tháng, không SLA rõ ràng.

Trade-off: Reliability cao hơn nhưng vận hành scheduler/worker phức tạp hơn cron đơn giản.

Ví dụ production: Pipeline daily 20 task + backfill + alert Slack → bắt buộc orchestration.

BigQuery

Dùng khi nào: Analytical query lớn, nhiều người dùng SQL, cần managed warehouse.

Không dùng khi nào: Workload chủ yếu transactional OLTP hoặc dữ liệu cực nhỏ/ít query.

Trade-off: Scale nhanh, ít ops; đổi lại cần kỷ luật query để tránh cost spike.

Ví dụ production: Team BI cần dashboard theo giờ trên hàng tỷ rows → BigQuery phù hợp.

dbt

Dùng khi nào: SQL transformation nhiều lớp, cần test/docs/lineage, teamwork qua Git.

Không dùng khi nào: Chỉ vài query ad-hoc ngắn hạn, chưa cần lifecycle rõ ràng.

Trade-off: Maintainable hơn nhưng thêm project structure và release discipline.

Ví dụ production: 60+ model cho finance/reporting, cần test tự động trước khi publish mart.

Spark

Dùng khi nào: Batch compute lớn, dataset vượt giới hạn single machine, cần distributed processing.

Không dùng khi nào: Data nhỏ/vừa có thể xử lý bằng SQL engine hoặc pandas đủ nhanh.

Trade-off: Scale tốt nhưng debug/tuning/shuffle phức tạp và tốn compute.

Ví dụ production: Backfill 2 năm clickstream parquet vài TB → Spark job partitioned.

Kafka / Flink

Dùng khi nào: Cần near-real-time, event-driven, xử lý stateful stream.

Không dùng khi nào: SLA theo giờ/ngày, không cần real-time, team chưa sẵn sàng on-call stream.

Trade-off: Latency thấp nhưng vận hành phức tạp (offset, checkpoint, lag, replay).

Ví dụ production: Fraud detection cần cảnh báo trong vài giây → Kafka + Flink là hợp lý.

14 · Interview Prep

Câu hỏi phỏng vấn theo level

Click vào câu hỏi để xem gợi ý trả lời. Junior cần trả lời được tất cả. Senior cần có depth ở mỗi câu.

Infrastructure & Data Platform
Tại sao dùng Docker trong data pipeline? Junior
Trả lời Junior: "Đảm bảo môi trường nhất quán giữa local, staging, production. Reproducible builds."

Trả lời Senior thêm: "Isolate dependencies giữa các service. Dễ scale horizontally. Container orchestration (K8s) cho phép auto-scaling và self-healing. Nhưng cần trade-off: overhead so với bare-metal, complexity của container networking, image size management."
Terraform state file là gì? Tại sao quan trọng và có những rủi ro gì? Mid
State file lưu mapping giữa Terraform resources và actual infrastructure. Terraform dùng nó để biết cần apply change gì.

Rủi ro: State file có thể chứa sensitive data (passwords, keys). Nếu state bị out of sync với thực tế (ai đó manually thay đổi infra) → terraform plan/apply có thể làm điều sai. Giải pháp: remote state (GCS/S3), state locking (tránh 2 người apply cùng lúc), encrypt state.
Design một hệ thống ingestion cho 10TB/ngày từ 100 nguồn khác nhau Senior
Không có một đáp án đúng — interviewer muốn thấy thinking process:

1. Phân loại nguồn: batch (file, DB dump) vs streaming (API, event stream)
2. Landing zone (Bronze): S3/GCS, raw format, immutable
3. Ingestion layer: Airbyte/Fivetran cho SaaS sources, custom connectors cho proprietary
4. Reliability: retry logic, dead letter queue cho failed records
5. Observability: row count, latency, error rate per source
6. Cost: compression (Parquet > CSV), partition by source+date, lifecycle policy
7. Schema management: Schema Registry hoặc catalog (Glue, DataHub)
Data Warehouse & Modeling
Partition và Cluster trong BigQuery khác nhau thế nào? Junior
Partition: Chia table thành các phần nhỏ theo column (thường là date). Query với WHERE trên partition column → BigQuery chỉ scan partition đó → tiết kiệm cost.

Cluster: Sort data trong mỗi partition theo column(s). Query với WHERE/ORDER BY trên cluster column → đọc ít blocks hơn.

Dùng cả hai: Partition by date, cluster by user_id → query filter theo date AND user → cực kỳ hiệu quả.
Giải thích SCD Type 2 và khi nào dùng Mid
SCD Type 2: Khi attribute thay đổi, thêm row mới thay vì update. Giữ toàn bộ history.

Ví dụ: Customer thay địa chỉ → thêm row mới với effective_start_date = hôm nay, đánh is_current = false row cũ.

Dùng khi: cần audit trail, cần biết trạng thái tại thời điểm X trong quá khứ, báo cáo historical.
Không dùng khi: history không quan trọng, table quá lớn (SCD2 làm table phình rất nhanh), chỉ cần current state.
Pipeline của bạn suddenly produce wrong numbers. Làm sao debug? Senior
Đây là câu hỏi về systematic debugging — không phải technical knowledge:

1. Xác định scope: tất cả metrics sai hay chỉ một số? Từ bao giờ? (git blame, deployment log)
2. Check data lineage: số sai ở layer nào? Gold → Silver → Bronze → Source
3. Compare với expected: row count, sum, distinct count tại mỗi layer
4. Check recent changes: code deploy, schema change, source data format change
5. Isolate: reproduce với subset data, test từng transformation riêng lẻ
6. Root cause + fix + verify + prevent: thêm data quality check để catch sớm hơn lần sau
Streaming & Batch
Exactly-once semantics trong Kafka là gì? Có thực sự đạt được không? Senior
Exactly-once = mỗi message được xử lý đúng một lần, không bỏ sót, không duplicate.

Trong distributed system, đây là bài toán cực kỳ khó. Kafka đạt được ở producer level (idempotent producer + transactions). Nhưng end-to-end exactly-once (từ source đến sink) phụ thuộc vào cả consumer và destination system.

Thực tế: at-least-once + idempotent consumer thường đủ và dễ implement hơn. Chỉ cần exactly-once khi: financial transaction, billing, critical counting.
Khi nào dùng batch, khi nào dùng streaming? Mid
Batch: Latency chấp nhận được (giờ/ngày), data volume lớn, phức tạp transformation, cost là ưu tiên, historical reprocessing thường xuyên.

Streaming: Low latency yêu cầu (giây/phút), alerting real-time, user-facing features cần fresh data, event-driven architecture.

Senior thêm: Lambda architecture (cả hai), Kappa architecture (streaming only với reprocessing). Trade-off: streaming phức tạp hơn, harder to debug, more expensive per record.

  • Tối ưu query/job để giảm ít nhất 20–30% thời gian hoặc chi phí so với baseline.
  • Giải thích trade-off giữa ít nhất 2 design choices cho mỗi module chính.
  • Document được kiến trúc, lineage, vận hành on-call cơ bản để người khác tiếp nhận.
  • Build ingestion có pagination + rate-limit + retry/backoff và chứng minh resume sau fail.
  • Áp dụng BigQuery partition/cluster + materialized view để tối ưu dashboard workload.
  • Triển khai dbt tests + quality gates để chặn deploy khi dữ liệu critical lỗi.

Senior Candidate

Tiêu chí: design system, xử lý failure/scale/cost/observability, review code và dẫn dắt kỹ thuật.

  • Thiết kế được kiến trúc data platform cho multi-source, có SLA/SLO, observability và ownership rõ.
  • Xử lý được incident mô phỏng 3h sáng: triage, rollback/replay, RCA, prevention.
  • Lập được kế hoạch scale khi data volume x100 kèm cost guardrail cụ thể.
  • Review code pipeline của người khác và chỉ ra risk về reliability/security/cost.
  • Kể trọn project end-to-end trong phỏng vấn bằng số liệu đo được và quyết định kiến trúc đã chọn.
  • Vận hành được Terraform CI/CD với remote state locking và policy kiểm soát apply.
  • Triển khai dbt Slim CI + Spark unit testing + quality gates xuyên suốt pipeline.
  • Thiết kế streaming flow có exactly-once strategy, lag observability và schema evolution an toàn.
Cách dùng checklist: Junior = nền tảng vận hành, Mid = tự build + tối ưu, Senior Candidate = tư duy hệ thống + ownership.
